# BUG-260511.4: Grind picker leaks assignments onto every iterated block

## Status: Done
## Severity: High (uneven work distribution)
## Version: v2.5.4
## Found In: `NanobotSystem.Grinding.cs` (ServerTryGrinding picker)

## Description

`ServerTryGrinding`'s picker iteration called `AssignToSystem` (a **claim**,
not a check) on every block it examined on the way to finding a grindable
target:

```csharp
if (... && !targetData.Block.AssignToSystem(_Welder.EntityId))
{
    skippedByAssign++;
    continue;
}
```

`AssignToSystem` returns false only when the block is held by a *different*
BaR â€” otherwise it claims the block to this BaR (refreshing the TTL) and
returns true. So every block iterated past before the picker's `break`
landed on a grindable target became claimed to this BaR, including:

- Fully-dismounted orphan blocks (skipped after the IsFullyDismounted check
  in BUG-260511.2 / .3),
- Blocks past the closest valid target (high-priority blocks held back, etc.),
- Blocks rejected by `IsGridOverSystemLimit` are *not* in this set (the limit
  check runs earlier and `continue`s before the claim), so the leak is the
  intersection of "passed grid-limit + passed assignment + reached this
  point".

Each leaked claim sits on the block for `AssignmentTtlSeconds` (2-8 s),
blocking other BaRs from grabbing it. On a 58-BaR fleet this manifested as
work piling up on a small subset of BaRs while others sat idle â€” even
though there were plenty of "free" blocks visible in the cluster's target
list.

Welding (`ServerTryWelding:131-141`) already used the correct two-phase
pattern: `IsAssignedToOtherSystem` (pure read) during iteration, then
`AssignToSystem` (claim) once after the loop breaks on the chosen target.
Grinding was inconsistent.

## Fix

Mirror the welding pattern.

### Inside iteration: check, don't claim

```csharp
if (Mod.Settings.AssignToSystemEnabled
    && Settings.CurrentPickedGrindingBlock == null
    && targetData.Block.IsAssignedToOtherSystem(_Welder.EntityId))
{
    skippedByAssign++;
    continue;
}
```

Reads the cache without mutating it. Other BaRs' claims remain intact;
this BaR doesn't accumulate phantom claims on blocks it merely walked past.

### After picking: actually claim

```csharp
if (chosenGrindTarget != null
    && Mod.Settings.AssignToSystemEnabled
    && _Welder.IsWorking && _Welder.Enabled
    && Settings.CurrentPickedGrindingBlock == null)
{
    if (!chosenGrindTarget.Block.AssignToSystem(_Welder.EntityId))
    {
        // Race: another BaR claimed it between read and write.
        skippedByAssign++;
        chosenGrindTarget = null;
    }
}
```

Only the chosen target is claimed. The race between iteration's read and
this write is the small window where another BaR might grab the block â€”
we just drop the target for this cycle and let the next picker try again.

## Effect

- Each BaR holds at most one assignment per cycle (the block it actually
  intends to grind).
- The TTL-refresh path is preserved: as long as a BaR keeps picking the
  same block on successive cycles (which it will, given the target list
  is sorted and the picker picks the first valid), `AssignToSystem` runs
  every cycle and the TTL never expires for the in-progress block.
- Other BaRs see far more free blocks. On a 58-BaR fleet, total claims
  drop from "up to 58 Ă— N (where N is iteration depth)" to "exactly 58
  (one per BaR)". Work distribution is no longer hostile to BaRs that
  arrive late in the per-tick processing order.

## Related

- BUG-260511.2 / .3 introduced the `IsFullyDismounted` filter and the
  orphan-raze enqueue. Both still work the same way after this fix â€” the
  iteration just no longer leaks claims on the orphan blocks before
  enqueueing them.
- ServerTryWelding's existing two-phase pattern (`IsAssignedToOtherSystem`
  early + `AssignToSystem` on chosen target) is the model.
