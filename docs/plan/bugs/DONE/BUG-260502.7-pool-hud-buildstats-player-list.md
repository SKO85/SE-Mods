# BUG-260502.7: Pool HUD `BuildStats` player list
## Status: Done (BuildId 260511.57)
## Severity: Low
## Version: v2.5.4
## Found In: REVIEW-260502.1 (full Codex code review), finding #8

## Description

`HudHandler.BuildStats` (`Handlers/HudHandler.cs:506`) allocates a fresh `List<IMyPlayer>` on every call (line 583) just to read `playerList.Count`:

```csharp
var playerList = new List<IMyPlayer>();
MyAPIGateway.Players.GetPlayers(playerList);
s.PlayerCount = playerList.Count;
```

`BuildStats` runs only when DebugMode or profiling is active (server-side, periodic), so the impact is bounded. Still avoidable GC pressure during long debug/profiling sessions.

## Fix

Reuse a static pooled scratch list, mirroring the pattern already used in `Handlers/NetworkMessagingHandler.cs:156` (`_adminCheckPlayers`):

```csharp
private static readonly List<IMyPlayer> _statsPlayerScratch = new List<IMyPlayer>();
// ...inside BuildStats:
_statsPlayerScratch.Clear();
MyAPIGateway.Players.GetPlayers(_statsPlayerScratch);
s.PlayerCount = _statsPlayerScratch.Count;
_statsPlayerScratch.Clear();
```

`BuildStats` is called only on the server main thread, so no concurrency guard needed.

## Verification

`dotnet build` clean. Enable DebugMode, confirm HUD `PlayerCount` continues to match the live player count.
