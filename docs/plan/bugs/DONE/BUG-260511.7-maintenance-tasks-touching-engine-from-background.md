# BUG-260511.7: Periodic maintenance tasks touched engine APIs from background threads

## Status: Done
## Severity: High (latent — intermittent crashes / state corruption under load)
## Version: v2.5.4
## Found In: `Managers/PeriodicMaintenanceScheduler.cs`
## Reported by: Codex review

## Description

`PeriodicMaintenanceScheduler.Tick` dispatched **every** periodic task onto
`MyAPIGateway.Parallel.StartBackground`. Three of those tasks reach into
engine state that is not safe to call from background threads:

| Task | Engine call(s) |
|---|---|
| `_friendlyRebuild` (`FriendlyRelationsHandler.Rebuild`) | `IMyEntity.GetUserRelationToOwner` — engine faction state |
| `_ownership` (`GridOwnershipCacheHandler.Update`) | `MyAPIGateway.Entities.TryGetEntityById`, `IMyCubeGrid.Closed` / `.BigOwners` |
| `_safeZone` (`SafeZoneHandler.CleanupSafeZones`) | `MySafeZone.MarkedForClose` / `.Closed`; also shares the `_staleZoneKeys` scratch buffer with main-thread paths (per its own `REF-2` comment) |

Off-thread access to these APIs can intermittently throw or corrupt state
under load, turning periodic maintenance into a stability risk. The bug was
latent — no consistent reproducer, but exactly the class of thing that
surfaces only when you have lots of factions / entities and high tick load.

## Fix

Added a `MainThread` flag to the `PeriodicTask` struct. When set, `TryFire`
runs the task's `Work` inline on the main thread instead of dispatching
to background. Three tasks are now flagged as main-thread:

- `_friendlyRebuild` — was bundled with profiler flush; split into its
  own task so the cadence stays at 5 s for both pieces without dragging
  the file-I/O onto the main thread.
- `_ownership` — was unflagged.
- `_safeZone` — was unflagged.

Tasks that legitimately do background work are unchanged:

- `_ttlCleanup` — TTL caches are `ConcurrentDictionary` operations.
- `_profilerFlush` — plain .NET file I/O (`StreamWriter.Flush`); kept on
  the background pool so file blocking can't stall the tick.

## Cost

Three tasks now add a small spike to one tick every 5–10 s:

- `FriendlyRelationsHandler.Rebuild`: O(distinct_owners × N). Memory has
  `BUG-260502.4` flagged as a follow-up to extend the interval / hook
  faction events. Worst-case sample (60 BaRs, 60 owners) ≈ 3,600
  `GetUserRelationToOwner` calls = sub-millisecond to a few ms.
- `GridOwnershipCacheHandler.Update`: bounded by `MaxRefreshItems` per
  call and short-circuits on idle grids.
- `SafeZoneHandler.CleanupSafeZones`: just walks `Zones` + a few TTL
  cleanups, sub-millisecond.

If the rebuild cost grows enough to be a problem on populated worlds
the existing BUG-260502.4 plan (event-driven invalidation or 15 s
interval) is the next step. For now correctness > raw throughput.
