# BUG-260511.20: Stale BlockSystemAssigningHandler claims block recovery when MaxSystemsPerTargetGrid drops

## Status: Done
## Severity: High (functional — limit change still strands the fleet)
## Version: v2.5.4
## Found In: `Mod.cs` (`SettingsChanged`)
## Reported by: Player follow-up after BUG-260511.13 / BUG-260511.14 / BUG-260511.19

## Symptom

Even with BUG-260511.13 (release surplus lock-ons) and BUG-260511.14
(reset exhausted flags) in place, lowering `MaxSystemsPerTargetGrid`
20 → 2 still left the fleet idle ("stops doing anything").

## Root cause

The release path in BUG-260511.13 only calls
`BlockSystemAssigningHandler.ReleaseAllForSystem` for **BaRs whose
current grind block is on an over-limit grid**. Other BaRs' stale
TTL claims (up to 8 s old, accumulated rapidly at WorkSpeed=10) stay
in the cache.

When the picker re-runs after the limit change:
- `IsGridOverSystemLimit` passes (count just dropped to 0).
- `IsAssignedToOtherSystem` hits true for many candidate blocks because
  the lingering claims are still inside the TTL window.
- All blocks get `skipAssign`'d. Picker exhausts.
- Even the few BaRs the new limit would have allowed to claim end up
  parking.

## Fix

In `Mod.SettingsChanged`, when `MaxSystemsPerTargetGrid` changes in
either direction, clear the entire assignment cache after the
exhausted-flag reset:

```csharp
if (_lastMaxSystemsPerTargetGrid > 0
    && currentLimit > 0
    && currentLimit != _lastMaxSystemsPerTargetGrid)
{
    foreach (var pair in NanobotSystems)
    {
        if (pair.Value != null) pair.Value.ResetLoopExhaustedFlags();
    }

    // Clean-slate the claim cache so stale TTL entries don't block
    // recovery picks.
    BlockSystemAssigningHandler.Clear();
}
```

A claim cache `Clear()` already exists for `Mod.UnloadData` — we
just call it on this code path too.

## Cost

`ConcurrentDictionary.Clear()` is O(N) in the dictionary size — a few
hundred entries at most at peak — single-digit microseconds.
Pickers immediately repopulate the cache as they claim fresh blocks
on the next tick.

## Why this is safe to do on limit-up too

Raising the limit means more BaRs can claim per grid. The current
claims aren't blocking anything functionally, but clearing them
forces a clean redistribution — every BaR's next claim is its own.
That avoids edge cases where a BaR holds a stale claim on a block
already ground by someone else.

## Forward-compat

If we ever surface more "limit"-like settings (e.g. per-grid weld
limit, per-cluster cap), the same clean-slate pattern should apply.
The Cache.Clear() call here is the canonical recovery surface.
