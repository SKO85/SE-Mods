# BUG-260502.2: IsProtectedFromGrinding fails open and caches the open result
## Status: Done
## Severity: High (protection bypass — re-confirmed by REVIEW-260511.1 finding #2)
## Version: v2.5.4
## Found In: REVIEW-260502.1 (Codex code review) finding #2, re-flagged by REVIEW-260511.1 finding #2

## Description

`SafeZoneHandler.IsProtectedFromGrinding` (`Handlers/SafeZoneHandler.cs:488`) wraps its body in a broad `catch` that writes `false` (not protected) to `ProtectedFromGrindingCache` and returns `false` (lines 589-593). The cache TTL is 15 s, so once an exception occurs for a `(target, attacker)` pair the BaR is permitted to grind that block for 15 seconds — even if the underlying issue clears within milliseconds.

Player impact: a transient SE error during a safe-zone evaluation can momentarily allow grinding on what should be a protected block, and the result of that evaluation is then locked in for 15 s.

## Root Cause

The catch was written as a "don't crash; let the BaR continue" guard but it propagates the *unsafe* assumption (block is grindable) into the cache. This is the inverse of the conservative choice: in the absence of evidence the block IS grindable, treat it as protected.

## Fix

Two minimal changes inside the catch at lines 589-593:

1. Return `true` (treat as protected) instead of `false` — so the BaR skips the block this tick and re-evaluates next tick.
2. **Do not** call `SetIsProtectedFromGrinding(...)` — let the cache stay empty so the next call retries the full evaluation rather than reading back the fail-open decision.

```csharp
catch
{
    return true; // fail-closed: skip the block, re-evaluate next call
}
```

Cost: when an exception occurs, the BaR pauses grinding that one block until the next call resolves cleanly. Vastly preferable to silently bypassing protection.

## Verification

Inject a forced exception in `GetIntersectingSafeZone` (e.g. throw conditionally on a known target id) and confirm grinding pauses on that target rather than proceeding.

## Resolution (BuildId 260511.45)

Implemented exactly as proposed:

```csharp
catch (Exception ex)
{
    Logging.Instance.Write(Logging.Level.Error, "SafeZoneHandler.IsProtectedFromGrinding: {0}", ex.Message);
    return true; // fail closed
}
```

Cache write removed. The next call retries the full evaluation against live engine state. Added an Error-level log so transient failures are visible during diagnosis (the original catch was silent).
