# BUG-260522.1: RazeQueueHandler drain budget bypassed by stale entries

## Status: Done (BuildId 260522.2)
## Severity: Medium (frame-spike risk under common post-combat cleanup)
## Version: v2.5.4
## Found In: `Handlers/RazeQueueHandler.cs`
## Reported by: Codex (chatgpt-codex-connector) — PR #136 review comment

## Description

`RazeQueueHandler.Process()` is supposed to cap per-tick work via
`MaxRazesPerDrainDefault` (default 5). It does this by gating the
`while` loop on `processed < MaxRazesPerDrainDefault`, but
`processed++` only happens **after** the dequeued block passes every
validity check and gets appended to `_batchByGrid`.

Stale entries — null target, missing grid, closed FatBlock, re-welded
(`!IsDestroyed`) — `continue` past the increment. So a single
`Process()` call can dequeue an arbitrary number of stale entries
without counting them toward the budget. Under common cleanup
conditions (large stale queue after combat/repairs, or after a long
quiet period drained TTL claims), that loop could monopolize the main
thread and produce a frame spike.

## Fix

Increment `processed` immediately after `_queue.Dequeue()`, before any
`continue`. Every dequeue counts toward the budget regardless of
outcome; stale entries no longer get a free pass.

```csharp
var entry = _queue.Dequeue();
processed++;

_pendingKeys.Remove(entry.Key);
// ...validity checks below now respect the budget via the bumped counter
```

`drained` (the success-only counter logged in the profiler) is kept
separate from `processed` so the profile output still distinguishes
"how many got razed" from "how many we looked at".

## Cost

None — the change makes the loop cheaper (early-exit at the budget
even when entries are mostly stale).
