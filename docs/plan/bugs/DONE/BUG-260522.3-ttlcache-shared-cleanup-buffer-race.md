# BUG-260522.3: TtlCache `_expiredKeysBuffer` race after BUG-260511.7/.18 split cleanup tasks

## Status: Done (BuildId 260522.2)
## Severity: Low (latent — would need a single cleanup task to overrun its scheduling interval)
## Version: v2.5.4
## Found In: `Caches/TtlCache.cs`
## Reported by: Codex (chatgpt-codex-connector) — PR #136 review comment

## Description

`TtlCache<TKey, TValue>.CleanupExpired()` previously reused a
per-instance `_expiredKeysBuffer` field across calls (the REF-2
optimisation) on the documented assumption that "All known
CleanupExpired call sites run on the main thread."

That assumption no longer holds. BUG-260511.7 / BUG-260511.18 left
several caches' cleanup paths on the background pool with no in-flight
guard in `PeriodicMaintenanceScheduler.TryFire`:

- `BlockSystemAssigningHandler.Cache` via `_assignmentCleanup` (5 s)
- `BlockFailureCooldownHandler.Cache` via `_ttlCleanup` (2 min)
- `InventoryHelper.ConnectionCache / TransferCache` via `_ttlCleanup` (2 min)
- `BlockPriorityHandling.GetItemKeyCache` via `_ttlCleanup` (2 min)
- `SharedGridBlockCache.Cache / SharedEntityCache.Cache` via `_ttlCleanup` (2 min)

If one of these background tasks ever overruns its scheduling interval
(huge cache + slow tick + scheduler quirk), the next fire would
dispatch a second worker that mutates the same `_expiredKeysBuffer`
List concurrently with the first — corrupting the removal set or
throwing `InvalidOperationException` and leaving expired entries in
place.

Practical impact: near-zero today (cleanups are sub-millisecond on
small caches). But the structural risk remains and the optimisation
isn't worth a single-digit-microsecond allocation savings.

## Fix

Switched to a method-local `List<TKey>` allocated lazily on first
expired entry encountered:

```csharp
public void CleanupExpired()
{
    TimeSpan now;
    if (!TryGetNow(out now)) return;

    List<TKey> expiredKeys = null;
    foreach (var pair in Entries)
    {
        if (pair.Value.IsExpired(now))
        {
            if (expiredKeys == null) expiredKeys = new List<TKey>();
            expiredKeys.Add(pair.Key);
        }
    }
    if (expiredKeys == null) return;

    CacheItem removed;
    for (int i = 0; i < expiredKeys.Count; i++)
        Entries.TryRemove(expiredKeys[i], out removed);
}
```

Lazy allocation means cleanups with no expired entries (the common
case for active caches) allocate nothing at all.

## Cost

- Common path (no expired entries): zero allocation.
- Expired-entry path: one `List<TKey>` allocation per cleanup call.
  Cleanups fire every 5 s to 2 min depending on cache → at most a
  handful of small List allocations per minute server-wide.
  Negligible.

## Alternative not taken

`PeriodicMaintenanceScheduler.TryFire` could have grown an
in-flight guard so the same task can't dispatch twice. That's more
comprehensive (protects future tasks too) but bigger surface area for
a low-frequency race. The TtlCache-local fix is the smaller change.
The scheduler guard can still be added later if some other shared
state surfaces.
