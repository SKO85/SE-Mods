# BUG-260511.15: Only the cluster coordinator works newly-discovered targets until the next member-poll interval

## Status: Done
## Severity: Medium (UX — long lag until full-fleet engagement after a new grid is scanned)
## Version: v2.5.4
## Found In: `NanobotSystem.Scanning.cs` (`UpdateSourcesAndTargetsTimer`, `AsyncClusterScan`, `AsyncApplyClusterResults`)
## Reported by: Player report

## Symptom

Player set `StaggerGroupCount = 1` (all BaRs should fire every cycle).
Pasted a damaged grid into a cluster's working area. The cluster scan
ran and found the new targets — but only **one BaR** (the cluster
coordinator) actually started grinding, working blocks one by one.
The other ~57 members sat idle until the player toggled
`StaggerGroupCount = 2` and back to `1`, at which point the whole
fleet engaged in parallel. On the next scan that discovered new
targets, the same one-BaR-only behaviour returned.

## Root cause

The cluster coordinator and the non-coordinator members run scan
publish / scan apply on *independent* timers:

- Coordinator's `AsyncClusterScan` publishes `cluster.SetResult(result)`,
  then immediately calls `ApplyClusterResultToSelf` so the coordinator's
  per-BaR `State.PossibleGrindTargets` is fresh.
- Members run `AsyncApplyClusterResults` only when their **own**
  `_LastTargetsUpdate` is older than `effectiveTargetInterval`
  (10 s active, 20 s in idle backoff).

If the cluster had been idle, member `_LastTargetsUpdate` values may
have been bumped quite recently by the same idle-cycle skip path
(`coordinator._scanSkippedSaturated`), so the next member apply was
many seconds away. Worst case: members were 9–19 seconds behind
the coordinator's fresh result; if subsequent scans hit
`IsTargetListSaturated()` and were skipped, members never re-applied
the new result at all.

The toggle "fix" the player discovered is exactly what
`TriggerImmediateRescan` does: it sets the firing BaR's
`_LastTargetsUpdate = TimeSpan.Zero` and `_rescanForced = true`. When
`Mod.SettingsChanged` fan-outs through every BaR's per-instance
`SettingsChanged`, every member's `_LastTargetsUpdate` gets reset to
zero → all members trigger `AsyncApplyClusterResults` on the next
tick → fleet syncs to the fresh result. That synchronisation should
have been automatic on a new publish, not require a manual toggle.

## Fix

`ScanClusterResult.Timestamp` is already set every publish. Members
now track the timestamp of the result *they actually applied* and
trigger an apply whenever the published timestamp is newer:

1. New field `NanobotSystem._lastAppliedResultTimestamp` (TimeSpan,
   init `TimeSpan.Zero`).
2. Set on every successful apply path:
   - `AsyncClusterScan` (coordinator) — right after its own
     `ApplyClusterResultToSelf`.
   - `AsyncApplyClusterResults` (member) — right after its
     `ApplyClusterResultToSelf`.
3. `UpdateSourcesAndTargetsTimer` extends the `updateTargets`
   condition: members also fire when
   `cluster.GetResult().Timestamp > _lastAppliedResultTimestamp`,
   bypassing the poll interval.

```csharp
if (!updateTargets && !cluster.IsCoordinator(this))
{
    var publishedResult = cluster.GetResult();
    if (publishedResult != null && publishedResult.Timestamp > _lastAppliedResultTimestamp)
    {
        updateTargets = true;
    }
}
```

After the fix, when the coordinator publishes a fresh result, the
next `Mod.UpdateBeforeSimulation` tick (every 2 s by default after
the recent tuning) sees `published > applied` on every member and
fires their `AsyncApplyClusterResults`. Fleet engagement lag drops
from up to 20 s (idle backoff) to at most 2 s (the Mod-level
`SourcesAndTargetsUpdateTimerInterval`).

## Why this stays cheap

- One TimeSpan compare per BaR per Mod-tick — negligible.
- The condition only fires when the timestamps actually differ, so
  steady-state ticks (no new scan) are a no-op.
- Members still respect `coordinator._scanSkippedSaturated`: if the
  coordinator skipped its scan, no new timestamp is published, no
  apply is forced. So FEAT-075's saturated-skip benefit is preserved.

## Not addressed here

The original FEAT-071 idle-backoff timing on members is unchanged.
That stretch was originally chosen to save background-thread cost on
idle fleets and is still useful — it only matters in the
*not-newly-published* case, which the new bypass doesn't touch.
