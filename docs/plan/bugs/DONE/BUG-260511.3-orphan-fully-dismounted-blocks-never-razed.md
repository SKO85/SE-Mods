# BUG-260511.3: Orphan fully-dismounted blocks never enqueued for raze

## Status: Done
## Severity: Medium
## Version: v2.5.4
## Found In: `NanobotSystem.Grinding.cs` (ServerTryGrinding picker), `Handlers/RazeQueueHandler.cs`

## Description

Follow-up to BUG-260511.2. After fixing the picker to use `IsFullyDismounted`
instead of `IsDestroyed`, blocks that had been left with integrity 0 *and*
stockpile empty (e.g. a wheel block on a detached subgrid after the suspension
was destroyed) were correctly skipped by the picker — but they still weren't
removed from the grid. `RazeQueueHandler.Enqueue` is only ever called from
inside `ServerDoGrind`, and `ServerDoGrind` doesn't run on picker-skipped
blocks, so the BaR's cleanup path never reached them.

Player reported: skeleton wheel blocks at 0 % integrity persist across grinds;
restarting the session lets the BaR pick up *some* of them again (because those
still had components — `IsFullyDismounted = false`) but the truly-empty ones
stayed forever.

## Fix

### 1. Picker enqueues orphan blocks for raze

`ServerTryGrinding` picker now enqueues fully-dismounted blocks for raze in
the else branch (where it previously just incremented `skippedByDestroyed`):

```csharp
else
{
    skippedByDestroyed++;
    RazeQueueHandler.Enqueue(targetData.Block);
}
```

The block isn't picked as a grind target (no work left), but the cleanup path
now sees it and `RazeBlocks` removes it from the grid.

### 2. Raze queue dedup

With 58 BaRs all iterating the same target list, the picker enqueue would
push the same orphan block dozens of times before `RazeQueueHandler.Process`
drained it (drain rate is 5 per 10 ticks). Added a `HashSet<BlockKey>` dedup
in `Enqueue`:

```csharp
private static readonly HashSet<BlockSystemAssigningHandler.BlockKey> _pendingKeys;

public static void Enqueue(IMySlimBlock block)
{
    if (block == null) return;
    var grid = block.CubeGrid;
    if (grid == null) return;
    var key = new BlockSystemAssigningHandler.BlockKey(grid.EntityId, block.Position);
    if (!_pendingKeys.Add(key)) return;
    _queue.Enqueue(block);
}
```

`Process` removes the key when it dequeues so the slot is freed for re-enqueue
if the engine refuses the raze (e.g. block re-welded between enqueue and now).

`BlockKey` is the existing struct from `BlockSystemAssigningHandler` — same
identity used everywhere else for "the same physical block".

## Effect

- Skeleton orphans that previously sat at 0 % integrity indefinitely now get
  razed by the existing `RazeQueueHandler` flow within a few seconds of being
  spotted by any BaR.
- Queue depth stays bounded — duplicates are O(1) rejected at Enqueue.
- No extra cost on the happy path (`ServerDoGrind` → `Enqueue` for newly
  dismounted blocks) because the dedup set just tracks one entry per
  in-flight raze.

## Related

- BUG-260511.2 (picker `IsDestroyed` → `IsFullyDismounted`) is the precondition
  for this fix. Without it the orphan blocks weren't even in
  `PossibleGrindTargets` long enough for the picker to see them — they were
  removed via the BUG-165 drop-from-list optimization.
- BUG-165 drop-from-list — preserved; uses `IsFullyDismounted` (matching the
  picker), so successful grinds that complete the block still drop it
  immediately.
