# BUG-260511.16: FEAT-076 fast-skip stays stuck when claims expire

## Status: Done
## Severity: High (functional — fleet engagement collapses to a fraction of available BaRs between cluster scans)
## Version: v2.5.4
## Found In: `NanobotSystem.cs`, `NanobotSystem.Grinding.cs`, `NanobotSystem.Welding.cs`
## Reported by: Player report (profile session `20260511220854-profiling`)

## Symptom

User pattern, even with `MaxGrindsPerTick = 100`,
`MaxGrindMsPerTick = 100`, and `MaxSystemsPerTargetGrid = 20`:

> "Starts with 1 or 2 grinders, then after a new scan arrives it boosts
> the grinding to many BaRs, but then again it goes down to a few."

Cyclic: every cluster scan, engagement bursts; between scans, it
collapses to a handful of BaRs.

## Profile evidence

Profile `20260511220854-profiling` (60 s, sim 1.0):

- `ServerDoGrind`: **2 427 calls / 60 s ≈ 40/s** → only ~5–6 BaRs
  actively grinding on average instead of the ~20 the limit allows.
- Picker log: 56 of 58 BaRs entries show
  `iterations=0; loopSkipped=True`. Two BaRs show
  `iterations=40; skipAssign=38; skipDestroyed=1`.

`skipAssign=38` is the smoking gun: 38 of 256 target blocks are
currently in `BlockSystemAssigningHandler` even though only 2 BaRs
are actively grinding. The remaining ~36 are stale TTL claims left
behind by BaRs that have since moved on (or completed blocks).

## Root cause

FEAT-076 fast-skip invalidates only on **two** signals:

1. Target-list `CurrentHash` changes (a new cluster scan publishes).
2. Per-BaR `_gridSaturation.Count` changes.

Neither moves when:

- A `BlockSystemAssigningHandler` TTL claim expires.
- A BaR explicitly calls `ReleaseFromSystem`.
- `GridSystemCount` drops because another BaR migrated grids.

So the picker correctly notes "all 256 blocks are claimed or
grid-limited right now" and parks on `_grindLoopExhausted = true`.
Over the next few seconds, ~8 s of claims drain out of the TTL cache,
slots open up — but the exhausted BaRs never re-iterate because the
hash hasn't moved. They wait for the next cluster scan (~5–10 s)
to lift exhaustion, by which time the same accumulate-then-collapse
cycle starts over.

## Fix

Add a third invalidation signal: **time**. An exhausted BaR retries
its full iteration after `ExhaustedRetryInterval`, regardless of
hash / saturation state.

```csharp
private static TimeSpan ExhaustedRetryInterval
{
    get { return TimeSpan.FromSeconds(Math.Max(1, Mod.Settings.AssignmentTtlSeconds / 3.0)); }
}
```

Interval is a fraction of `AssignmentTtlSeconds` (default 8 s → retry
every ~2.7 s) so a BaR re-checks at least three times per claim
lifetime. The interval auto-adjusts if the admin changes
`AssignmentTtlSeconds`.

Implemented on both `_grindLoopExhausted` (`NanobotSystem.Grinding.cs`)
and `_weldLoopExhausted` (`NanobotSystem.Welding.cs`). Each picker
records `_grindExhaustedAtTime` / `_weldExhaustedAtTime` when it
goes exhausted, and the fast-skip gate checks the age against
`ExhaustedRetryInterval`.

```csharp
var grindExhaustedAge = Mod.NowPlayTime.Subtract(_grindExhaustedAtTime);
if (_grindLoopExhausted
    && State.PossibleGrindTargets.CurrentHash == _grindExhaustedAtHash
    && _gridSaturation.Count == _grindExhaustedSaturatedCount
    && grindExhaustedAge < ExhaustedRetryInterval)
{
    // fast-skip
}
```

## What this does NOT do

- It does not eliminate the fast-skip path. When no claims have
  expired and the BaR genuinely has no work, the fast-skip still
  saves the iteration cost for `ExhaustedRetryInterval` worth of
  ticks.
- It does not change AssignmentTtlSeconds; admins still tune that.
- It does not address the underlying stale-claim accumulation
  (BaRs leaving claims on completed blocks). That's a separate
  potential follow-up — explicit release-on-completion in the
  grinder loop, similar to what welding does.

## Expected impact

The cyclic "1-2 BaRs → boost → 1-2 BaRs" pattern should flatten. BaRs
that exhaust still re-engage within ~2–3 s as claims drain, instead of
waiting for the next cluster scan. Fleet engagement should approach
the `MaxSystemsPerTargetGrid` limit steadily, not in bursts.
