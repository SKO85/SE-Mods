# BUG-260511.5: Solo-cluster stagger too aggressive for low-BaR-count operations

## Status: Done
## Severity: Low (UX / perceived speed, no functional issue)
## Version: v2.5.4
## Found In: `NanobotSystem.Update.cs` (`UpdateBeforeSimulation10_100` stagger calc)

## Description

Player report: with only 1 or 2 BaRs actively grinding (in a world that has
many other placed BaRs), the active BaR felt sluggish.

## Root cause

The per-BaR effective stagger for solo clusters (`clusterSize == 1`) was
copied directly from the mod-wide value, which auto-scales by **total**
placed BaR count:

| Total placed BaRs | Mod-wide stagger |
|---:|---:|
| ≤ 5 | 1 |
| ≤ 10 | 2 |
| ≥ 11 | 3 |

So a single BaR doing work in a world with 11+ placed BaRs (most of them
idle or disabled) inherited stagger 3 — meaning it fired only every 3rd
cycle. At `WorkSpeed = 10` that's 2 fires/sec instead of 6 — visibly slow
for an operation the player is watching.

The original BUG-102 fix introduced this scaling to prevent CPU spikes when
many isolated BaRs all fire on the same tick. The reasoning is sound, but
3-way splitting is overly aggressive for the typical "small fleet at work"
case — the CPU saving from 3-way vs. 2-way staggering of solo BaRs is minor
unless the world has many dozens of isolated BaRs.

## Fix

Cap the solo-cluster effective stagger at 2:

```csharp
if (clusterSize == 1)
{
    effectiveGroups = Math.Min(modWideStagger, 2);
}
```

Behaviour matrix after the change:

| Total placed BaRs | Solo-cluster stagger | Cluster 2-5 | Cluster ≥ 6 |
|---:|---:|---:|---:|
| ≤ 5 | 1 (unchanged) | 1 (unchanged) | n/a |
| ≤ 10 | 2 (unchanged) | 1 (unchanged) | 2 capped |
| ≥ 11 | **2** (was 3) | 1 (unchanged) | up to 3 |

So:
- A single working BaR among many idle/disabled BaRs now fires every other
  cycle instead of every 3rd. Roughly 1.5× the perceived responsiveness.
- Small clusters (2-5) are unaffected — they already collapsed to 1.
- Large clusters (6+) are unaffected — they cap at mod-wide (3 max).

Admin override (`/nanobars config set StaggerGroupCount N`) still wins
over the auto-scaling, so admins who want full mod-wide stagger for solo
BaRs can set `StaggerGroupCount=3` explicitly.

## Why not lower the mod-wide cap globally

Tempting, but the mod-wide value is also used for the cluster-≥-6 path
(`effectiveGroups = Math.Min(modWideStagger, clusterSize - 3)`). Large
clusters genuinely benefit from 3-way staggering of their per-tick
`ApplyClusterResultToSelf` filter cost. The scope of this fix is solo
clusters only, where the per-tick cost is dominated by one BaR's own
scan and the work is naturally serialised.
