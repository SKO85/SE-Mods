# BUG-260511.8: State fingerprint missed lock-on target changes

## Status: Done
## Severity: High (client state divergence — stale visuals on weld/grind targets)
## Version: v2.5.4
## Found In: `NanobotSystem.Operations.cs` (`ComputeStateFingerprint`)
## Reported by: Codex review

## Description

`TryTransmitState` short-circuits when the computed state fingerprint
matches the last-transmitted fingerprint (the `BUG-150` fpUnchanged path),
resetting `State.Changed` and skipping the send.

But `ComputeStateFingerprint` did **not** include
`State.CurrentWeldingBlock` / `CurrentGrindingBlock`. Their setters DO
flip `State.Changed = true` when the physical block changes, so
`IsTransmitNeeded()` correctly reported "send needed" — only for the
fingerprint compare to immediately suppress that send.

Effect on clients: lock-on target changes (BaR switches from block A to
block B) were silently dropped whenever no other hashed field also
changed in the same window. Beam/weld effects and any UI tied to the
current target stayed on block A until some unrelated field (a target
list hash, an Inventory* flag, etc.) shifted.

## Fix

Hash both lock-on blocks into the fingerprint. Key derived from
`CubeGrid.EntityId + Position` — same identity tuple the setters use
to decide whether the value is a "same physical block" reference
refresh (which doesn't flip `Changed`). So a same-physical-block
re-assignment leaves the fingerprint identical and the suppression is
still correct; a true target swap flips the fingerprint and triggers
the send.

```csharp
hash = hash * 31 + ComputeBlockKey(State.CurrentWeldingBlock);
hash = hash * 31 + ComputeBlockKey(State.CurrentGrindingBlock);

private static long ComputeBlockKey(IMySlimBlock block)
{
    if (block == null) return 0L;
    var gridId = block.CubeGrid != null ? block.CubeGrid.EntityId : 0L;
    var pos = block.Position;
    long key = gridId;
    key = key * 31 + pos.X;
    key = key * 31 + pos.Y;
    key = key * 31 + pos.Z;
    return key;
}
```

No allocation, no engine call.

## Why this regression slipped past BUG-150

BUG-150's intent was "skip the send when nothing visible changed." The
fingerprint covered every hashed flag plus the four target-list / missing-
component hashes, which catches almost everything — but the lock-on
target lives on `State` directly (single block reference per side), not
as a list, so it was missed. The `Changed` flag was the only other
signal that would have caught it, and the fpUnchanged path discards
`Changed` before the send.
