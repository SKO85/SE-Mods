# BUG-260511.6: Auto `StaggerGroupCount` counts disabled BaRs

## Status: Done
## Severity: Low (UX / perceived speed, no functional issue)
## Version: v2.5.4
## Found In: `Mod.cs` (`GetEffectiveStaggerGroupCount`)

## Description

Player report: world had 2 enabled BaRs + 58 disabled BaRs (leftovers from a
fleet test). `/nanobars systems count` correctly reported only the 2 active
ones, but the debug HUD's `Stagger:` line and the auto formula returned **3**
because the total placed-block count was 60.

Effect on the active fleet was small (the per-BaR `effectiveGroups` math
already collapses 2-member clusters to 1), but the displayed number was
misleading and the formula's own intent — scale staggering to the working
population — wasn't being met.

## Root cause

`GetEffectiveStaggerGroupCount` counted every entry in `NanobotSystems`
(the ConcurrentDictionary of `MyGameLogicComponent` instances). That
collection includes disabled BaRs, which short-circuit out of
`ServerTryWeldingGrindingCollecting` and contribute no work — they should
not influence the auto bucket.

## Fix

Count only entries where `_Welder.Enabled == true`:

```csharp
var enabled = 0;
foreach (var kvp in NanobotSystems)
{
    if (kvp.Value != null && kvp.Value.IsEnabled) enabled++;
}
if (enabled <= 5) return 1;
if (enabled <= 10) return 2;
return StaggerGroupCountDefault;
```

Added a small public accessor `NanobotSystem.IsEnabled` (wraps
`_Welder != null && _Welder.Enabled`) because `_Welder` is private.

## Caching

`GetEffectiveStaggerGroupCount` is called from each BaR's `Update10`
(plus the HUD), so on a 60-BaR world that's ~360 calls/sec.
ConcurrentDictionary enumeration is cheap but allocates an enumerator
per call → unnecessary GC pressure.

Added a tiny per-frame-counter cache (~1 s TTL @ 60 Hz) keyed by
`MyAPIGateway.Session.GameplayFrameCounter`:

```csharp
private const int StaggerAutoCacheFrames = 60;
private static int _cachedAutoStagger = -1;
private static int _cachedAutoStaggerFrame = int.MinValue;
```

Flipping a BaR on/off shows the new bucket within ~1 s. No event hook
needed; the staleness is below the typical "I just toggled some BaRs and
look at the HUD" feedback latency.

Explicit override (`Settings.StaggerGroupCount > 0`) still bypasses both
the count and the cache, so admin overrides are immediate.
