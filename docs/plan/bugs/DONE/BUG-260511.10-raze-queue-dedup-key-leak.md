# BUG-260511.10: Raze queue dedup keys leaked when block lost its grid

## Status: Done
## Severity: Medium (slow leak — `_pendingKeys` grows, future enqueues for same blocks silently rejected)
## Version: v2.5.4
## Found In: `Handlers/RazeQueueHandler.cs`
## Reported by: Codex review

## Description

`RazeQueueHandler.Process` removed each block's dedup key from
`_pendingKeys` mid-loop, **after** the null-grid check:

```csharp
var target = _queue.Dequeue();
if (target == null) { skippedNullTarget++; continue; }     // never frees key
var grid = target.CubeGrid;
if (grid == null) { skippedNullGrid++; continue; }         // never frees key
_pendingKeys.Remove(new BlockKey(grid.EntityId, target.Position));   // only path that frees the key
```

When a block was enqueued with a valid grid and then the grid was
removed (detached subgrid, world cleanup, raze of the parent block by
something else) before `Process` got to it, the dequeued
`target.CubeGrid` is null. The loop `continue`s to the next entry, but
`_pendingKeys` still holds the entry's key. Worse, the key was
constructed from `grid.EntityId` at enqueue — without `grid` at
dequeue we can no longer reconstruct the key from `target` alone, so
the original removal call was structurally unable to clean up.

Effects:

- `_pendingKeys` grows monotonically during heavy grid removal
  activity (combat, world cleanup, mass dismount).
- Once the leak runs long enough, future `Enqueue` calls for blocks
  at the same grid+position get silently rejected by the `_pendingKeys.Add`
  short-circuit, so those orphans never get razed (relevant for ships
  that have been damaged/rebuilt at the same coordinates).

## Fix

Pair the block with its dedup key at enqueue time and store both in a
`QueueEntry` struct:

```csharp
private struct QueueEntry
{
    public BlockSystemAssigningHandler.BlockKey Key;
    public IMySlimBlock Block;
}
```

`Process` then removes `entry.Key` from `_pendingKeys` **before** any
`continue` path runs, so every dequeued entry frees its slot regardless
of `target.CubeGrid` state.

```csharp
var entry = _queue.Dequeue();
_pendingKeys.Remove(entry.Key);   // unconditional, before any skip path

var target = entry.Block;
if (target == null) { skippedNullTarget++; continue; }
...
```

Net code change: `Queue<IMySlimBlock>` → `Queue<QueueEntry>`,
`_queue.Enqueue(block)` → `_queue.Enqueue(new QueueEntry { … })`,
key-removal moved to the top of the loop body. Memory cost is
2 longs + 3 ints per pending entry — negligible vs. the
`IMySlimBlock` reference already held.
