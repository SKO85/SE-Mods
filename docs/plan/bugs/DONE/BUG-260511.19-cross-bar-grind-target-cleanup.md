# BUG-260511.19: At WorkSpeed=10, only 1 BaR grinds because cross-BaR completions stay in everyone else's list

## Status: Done
## Severity: High (the WorkSpeed-dependent "1 BaR grinding while 19 sit idle" pattern)
## Version: v2.5.4
## Found In: `NanobotSystem.cs` (new helper), `NanobotSystem.Grinding.cs`
## Reported by: Player observation
> "All BaRs cover the area. At WorkSpeed=10 only 1 BaR grinds, one by one. At WorkSpeed=1 all 20 grind."

## Why WorkSpeed=10 vs WorkSpeed=1 changes the outcome

`BUG-165` removes a block from `State.PossibleGrindTargets` only when **this**
BaR finishes grinding it. Cross-BaR completions stay in every other member's
local list as `FatBlock.Closed` entries until the next cluster-scan publish
rebuilds (5–10 s).

The dead-entry accumulation rate is `~6 × N` per second where N is the number
of actively grinding BaRs. At WorkSpeed=10 with 20 BaRs grinding, that's
~120 closed entries/s spreading into each of the other 57 members' lists.
Within a single scan interval, a typical 256-block list fills almost
entirely with closed entries; the picker iterates past them
(`skipClosed++`), exhausts, and parks.

At WorkSpeed=1 the rate is **1/10**, so dead entries accumulate slowly
enough that members still find live work on every picker tick.

This is also what made BUG-260511.17 (the post-iteration `RemoveAll` prune)
look like the right fix — and it would have helped at WorkSpeed=10. The
reason that prune showed lower `ServerDoGrind` in its profile was a
different scenario (cluster scan returning empty results) — not a
contradiction of the underlying analysis.

## Fix

At the **source** of each block completion (the BaR that just ground it),
propagate the removal across all cluster members. Two parts:

### 1. New helper `NanobotSystem.RemoveGrindTargetByPosition`

Matches by `CubeGrid.EntityId + Position` (NOT reference) — each member's
`ApplyClusterResultToSelf` wraps cluster candidates in fresh
`TargetBlockData` instances, so reference equality is unreliable across
BaRs.

```csharp
internal void RemoveGrindTargetByPosition(long gridEntityId, Vector3I position)
{
    lock (State.PossibleGrindTargets)
    {
        for (int i = State.PossibleGrindTargets.Count - 1; i >= 0; i--)
        {
            var t = State.PossibleGrindTargets[i];
            if (t == null || t.Block == null || t.Block.CubeGrid == null) continue;
            if (t.Block.CubeGrid.EntityId == gridEntityId
                && t.Block.Position.X == position.X
                && t.Block.Position.Y == position.Y
                && t.Block.Position.Z == position.Z)
            {
                State.PossibleGrindTargets.RemoveAt(i);
                State.PossibleGrindTargets.ChangeHash();
                return;
            }
        }
    }
}
```

### 2. Call it from the grind completion path

Inside `ServerTryGrinding`, right after the existing `BUG-165` self-prune
when `IsFullyDismounted`, iterate `AssignedCluster.Members` and call the
helper on each non-self member:

```csharp
var cluster = AssignedCluster;
if (cluster != null && grindBlock.CubeGrid != null)
{
    var gid = grindBlock.CubeGrid.EntityId;
    var pos = grindBlock.Position;
    var members = cluster.Members;
    for (int m = 0; m < members.Count; m++)
    {
        var other = members[m];
        if (other != null && other != this) other.RemoveGrindTargetByPosition(gid, pos);
    }
}
```

## Cost

Per block completion: `cluster.Members.Count - 1` lock acquires + list
walks. With 58 members and ~120 completions/s globally at WorkSpeed=10,
that's ~6 900 cross-BaR cleanup calls/s. Each is a list walk averaging
~128 entries → ~µs per call → roughly 8–10 ms/s total. Lives in the
grind tick budget but is small relative to the engine grind cost itself.

Both the grind path and `RebuildClusters` run on the main thread, so
the cluster member list snapshot is stable for the duration of the
iteration.

## Welding side

Welding has the same shape — a successful weld's effects don't propagate
to other members' target lists. Not addressed in this ticket because
the user's profile is grinding-only and weld blocks tend to ramp to
full integrity rather than vanish, so the closed-FatBlock issue is much
weaker there. If a future profile shows the same pattern on welds,
mirror this fix in `NanobotSystem.Welding.cs`.
