# BUG-260502.6: Remove dead `_lastClusterKey` field
## Status: Done (BuildId 260522.1)
## Severity: Low
## Version: v2.5.4
## Found In: REVIEW-260502.1 (full Codex code review), finding #7

## Description

`NanobotSystem._lastClusterKey` (`NanobotSystem.cs:341`) is a vestigial string field. It is written at:

- `Cluster/ScanClusterCoordinator.cs:82` — reset to `null` for excluded systems
- `Cluster/ScanClusterCoordinator.cs:88` — assigned the freshly computed cluster key

…and never read anywhere. FEAT-072 introduced `_lastClusterKeyHash` (an int) for the fast-path settings-change detection in `RebuildClusters`; the string field was kept around but no longer drives any decision.

## Fix

- Delete the `_lastClusterKey` field declaration in `NanobotSystem.cs`.
- Delete the two write sites in `ScanClusterCoordinator.cs` (lines 82 and 88).
- Update the docstring above the remaining `_lastClusterKeyHash` if it still references the string version.

Trivial cleanup. Reduces per-instance memory and removes a confusing "is this still load-bearing?" footgun for future readers.

## Verification

`dotnet build` clean. Run a session, toggle a `ClusterRelevantFlags` setting, confirm cluster reshuffle still happens (relies on `_lastClusterKeyHash`, unchanged).
