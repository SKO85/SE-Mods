# BUG-260511.14: `_grindLoopExhausted` stays stuck after `MaxSystemsPerTargetGrid` cycles back up

## Status: Done
## Severity: High (functional — fleet stays idle after a routine admin action)
## Version: v2.5.4
## Found In: `Mod.cs` (`SettingsChanged`), `NanobotSystem.cs` (`ResetLoopExhaustedFlags`)
## Reported by: Player report (verified via profile session `20260511213943`)

## Symptom

After applying BUG-260511.13 (which fixed the deadlock when lowering
`MaxSystemsPerTargetGrid`), a follow-up scenario stayed broken:

- Cycle the limit `20 → 1 → 20`, or `20 → 2 → 20`: **no BaR grinds
  anymore**, even though the limit is back to the original generous value.

Sim is OK (back at 1.0 — that part of BUG-260511.13 worked), so the
issue is purely functional.

## Profile evidence

Session `20260511213943-profiling` captured **while broken**:

```
domain=Grind;calls=10722;totalMs=9.811;avgMs=0.001;maxMs=0.053
method=ServerTryGrinding;calls=6960;totalMs=8.517;avgMs=0.001
method=ServerTryWelding;calls=6960;totalMs=11.182;avgMs=0.002
```

The pickers are being called every cycle (~116 calls/s across the
fleet) but each returns in ~1–2 µs. That signature is the FEAT-076
fast-skip firing on every entry — i.e. `_grindLoopExhausted == true`
and the guard conditions still match.

## Root cause

FEAT-076 fast-skip gate:

```csharp
if (_grindLoopExhausted
    && State.PossibleGrindTargets.CurrentHash == _grindExhaustedAtHash
    && _gridSaturation.Count == _grindExhaustedSaturatedCount)
{
    loopSkipped = true;
    return;
}
```

The trap:

1. At `limit = 1`, 55+ BaRs iterate the target list, find every block
   on a saturated grid, mark themselves exhausted with
   `_grindExhaustedSaturatedCount = _gridSaturation.Count`. But the
   saturation set uses **strict `>`** (`count > limit`), and with all
   grids having exactly 1 BaR in `GridSystemCount`, **`1 > 1` is
   false** → saturation set is empty → `_gridSaturation.Count = 0`.
2. So the exhausted snapshot is `count == 0`.
3. Admin raises the limit back to 20. BUG-260511.13's release path
   doesn't fire (limit went up). Per-BaR `SettingsChanged` fires
   `TriggerImmediateRescan`, but the rescan is **async**.
4. On the next picker tick (before the async scan completes):
   - `_grindLoopExhausted` is still `true`.
   - `_gridSaturation.Count` is still `0` (3 grids have count=1, not
     `> 20`).
   - `_grindExhaustedSaturatedCount` is `0` from step 1.
   - `CurrentHash` is unchanged (scan hasn't republished yet).
   - **All three guards match — fast-skip fires.**
5. The scan eventually completes and changes the hash, **but** if the
   block list happens to be similar enough (BaRs hadn't released
   block claims, target lists barely change), the hash can match too
   in edge cases.

Even setting the hash issue aside: the count-guard collision is a real
correctness gap — `_grindExhaustedSaturatedCount` was meaningful only
under the limit that produced it, not under a *different* limit.

## Fix

Two-part:

### 1. `NanobotSystem.ResetLoopExhaustedFlags()` (new)

Clears both `_weldLoopExhausted` and `_grindLoopExhausted`. The
next picker entry then performs a full iteration regardless of the
stale snapshot values.

### 2. `Mod.SettingsChanged` extension

Whenever `MaxSystemsPerTargetGrid` **changes in either direction**
(up or down), iterate every BaR and call `ResetLoopExhaustedFlags`:

```csharp
if (_lastMaxSystemsPerTargetGrid > 0
    && currentLimit > 0
    && currentLimit != _lastMaxSystemsPerTargetGrid)
{
    foreach (var pair in NanobotSystems)
    {
        if (pair.Value != null) pair.Value.ResetLoopExhaustedFlags();
    }
}
```

Runs after the BUG-260511.13 lowered-limit release (so a downward
change releases lock-ons AND resets exhausted flags in one pass) and
before per-BaR `SettingsChanged` triggers the rescans.

## Why not invalidate per-BaR on each picker entry

The fast-skip is intentional — it saves the cost of iterating large
target lists when nothing has changed. Resetting only on
`MaxSystemsPerTargetGrid` change is the targeted fix: the guard's
implicit assumption is "saturated-count is meaningful under the same
limit value." When the limit changes, that assumption breaks.

## Forward-compat note

The same trap could appear with any other field that influences which
blocks the picker accepts (priority lists, search mode, owner change,
etc.). Most of those are already covered because they shift the
cluster-key hash, which flips the per-target-list `CurrentHash`. The
limit value is mod-wide and doesn't enter any cluster key — hence
this explicit reset.
