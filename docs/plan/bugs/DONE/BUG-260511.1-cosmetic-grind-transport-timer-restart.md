# BUG-260511.1: Cosmetic grind transport timer kept restarting

## Status: Done
## Severity: Medium (visual only — no game-state effect)
## Version: v2.5.4
## Found In: `NanobotSystem.Grinding.cs` (`ServerDoGrind` transport-gate block)

## Description

After multi-grind landed, players reported the particle-trail visualization for
grind transport "kept being sent out but never returned" — particles streamed
from the welded block toward the BaR but the animation never finished. Visually
the BaR looked stuck in transport mode forever.

## Root cause

The transport-gate block in `ServerDoGrind` unconditionally overwrote
`State.CurrentTransportStartTime = playTime` every time the gate opened
(transport inventory full *or* block fully dismounted). With the multi-grind
wrapper firing several `ServerDoGrind` calls in one cycle (each potentially
destroying a different block), the cosmetic timer got pinned to "now" on every
iteration — so `IsTransportRunning` never saw elapsed time ≥ `CurrentTransportTime`
and the visual animation never completed.

## Fix

Separate the *cosmetic* setup from the *functional* inventory transfer. Only
seed a fresh transport when none is already in flight; always do the inventory
empty:

```csharp
if (transportGateOpen)
{
    var transportAlreadyRunning = State.CurrentTransportStartTime > TimeSpan.Zero;

    if (!transportAlreadyRunning)
    {
        // Seed cosmetic transport
        State.CurrentTransportIsPick = true;
        State.CurrentTransportTarget = ComputePosition(target);
        State.CurrentTransportStartTime = playTime;
        State.CurrentTransportTime = TimeSpan.FromSeconds(2d * targetData.Distance / Settings.GrindTransportSpeed);
    }

    // Inventory transfer is functional, runs every gate-open
    ServerEmptyTransportInventory(true);
    transporting = true;
}
```

Subsequent grind operations in the same cycle keep the original animation running
to completion; only the inventory empty fires each time. After `IsTransportRunning`
finally resets `StartTime` to zero (cosmetic timer elapsed), the next gate-open
seeds a fresh transport.

## Note

The welding-side transport setup (`ServerFindMissingComponents` in
`NanobotSystem.Welding.cs`) already had this gate — `if (IsTransportRunning(playTime)) return true;`
at the head of the method short-circuits before any state write. Welding was
unaffected; the bug was grind-side only.

## Related

- FEAT-260511.6 (multi-action wrapper) was the trigger for this regression.
  With `MaxActionsPerCycle` later reduced to 1 the bug no longer surfaces in
  practice, but the fix is left in place — same code path is exercised by
  full-grid-grind scenarios where a single BaR finishes blocks in successive
  ticks.
