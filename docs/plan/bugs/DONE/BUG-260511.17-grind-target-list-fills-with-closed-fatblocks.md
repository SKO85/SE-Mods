# BUG-260511.17: Per-BaR grind target list fills with closed-FatBlock entries between cluster scans

## Status: Reverted (the fix did not help — see "Why reverted" at the bottom)
## Severity: High (the cyclic engagement collapse that BUG-260511.16's time-retry only partially fixed)
## Version: v2.5.4
## Found In: `NanobotSystem.Grinding.cs`
## Reported by: Player report + profile session `20260511221733-profiling`

## Symptom

Even after the BUG-260511.16 time-based retry fix, the user still saw
cyclic grind engagement: ~20 BaRs briefly, dropping to ~8, periodic
~20 spikes when a new cluster scan publishes, then drops again.

## Profile evidence

Profile `20260511221733-profiling` shows the retry path firing — but
the retries find empty lists:

```
iterations=256;skipNull=0;skipClosed=174;skipGrid=0;skipAssign=82;
   skipScript=0;skipDestroyed=0;loopSkipped=False
```

**174 of 256 target entries are `FatBlock.Closed`** — already-ground
blocks that should have been removed from the list but weren't.
Combined with the 82 claims, the entire list is unusable.

## Root cause

Each BaR maintains its own per-BaR `State.PossibleGrindTargets` —
populated by `ApplyClusterResultToSelf` from the shared cluster scan
result and filtered through that BaR's range OBB. The list is
**only refreshed by the next cluster scan** (~5–10 s).

BUG-165 removes a block from `State.PossibleGrindTargets` when THIS
BaR finishes grinding it (`IsFullyDismounted`). But a block ground
to completion by ANOTHER cluster member is **not** removed from this
BaR's local list. The other BaR's `RazeQueueHandler` eventually
destroys the FatBlock, but the dangling `TargetBlockData` wrapper in
this BaR's list still references the now-closed block.

In a 58-BaR cluster grinding ~20 blocks/s steady-state, ~5–10 s
between scans means **100–200 blocks completed elsewhere**, all
sitting in each non-grinding BaR's list as `FatBlock.Closed` entries.
The picker walks past them (`skipClosed++; continue;`), eventually
exhausts, and parks. The time-retry from BUG-260511.16 re-iterates
every 2.7 s — but every retry walks the same dead list, finds
nothing, and re-exhausts. Engagement only recovers when the next
cluster scan rebuilds the list fresh.

That's the cycle: scan → fresh list → ~20 grind → list fills with
closed entries from cross-BaR work → others can't find anything →
drops to ~8 BaRs (the ones whose own current-grinding-block is still
live) → next scan → repeat.

## Fix

After the foreach exits, if iteration counted ≥ 16 closed-FatBlock
skips, prune them from `State.PossibleGrindTargets` and bump the hash
so the next retry sees the cleaned list:

```csharp
if (skippedByClosedFatBlock >= 16)
{
    State.PossibleGrindTargets.RemoveAll(t =>
        t != null && t.Block != null && t.Block.FatBlock != null && t.Block.FatBlock.Closed);
    State.PossibleGrindTargets.ChangeHash();
}
```

Threshold of 16 gates the cost — a quiet cluster won't pay the
RemoveAll on every picker call. Under heavy cross-BaR grinding the
prune fires often, keeping the list lean.

Runs **inside** the existing `lock (State.PossibleGrindTargets)`, so
no extra synchronisation. `RemoveAll` is O(N); for 256 entries it's
~25 µs. Cheap relative to the alternative — continuing to walk dead
entries and stay exhausted.

## What the prune does NOT fix

- Stale `BlockSystemAssigningHandler` claims (the BUG-260511.16
  follow-up about releasing on block completion). Those still
  TTL-expire on their own and don't show up in this profile as the
  dominant cost.
- Welding has the same closed-FatBlock-skip pattern but no
  cross-BaR cleanup. The user's profile is grinding-only and we
  don't have welding-side evidence yet; if a future profile shows
  the same pattern there, mirror this fix in `NanobotSystem.Welding.cs`.

## Expected impact

Engagement should track the limit (~20 BaRs) much more steadily
between cluster scans, instead of collapsing as closed entries
accumulate. The "boost on scan, drop to few, boost again" cycle
should flatten significantly.

## Why reverted

Profile `20260511222829-profiling` (with the prune applied) showed
`ServerDoGrind` actually **dropped** from 2 004 calls / 60 s to
522 calls / 60 s vs the previous build — a 4× regression.

The prune itself does what it advertises (closed entries do get
removed), but the user's actual situation in the test session was
that the **cluster scan was returning very few candidates** (e.g.
`grindTargets=32(pre=32,grids=3)` and later `grindTargets=0`).
Whatever blocks existed had been ground, and the scan wasn't finding
new ones fast enough. The closed-FatBlock accumulation is real, but
fixing it doesn't matter when the underlying scan throughput is the
bottleneck.

Worse, with `RebuildHash` happening on apply but `ChangeHash` (no
count update) inside the prune, the `CurrentCount` field went stale
in confusing ways while debugging. Better to revert and look at the
actual scan-throughput problem.

The real cause of the cyclic engagement turned out to be elsewhere —
see the user's follow-up profile and BUG-260511.18 / future tickets.
