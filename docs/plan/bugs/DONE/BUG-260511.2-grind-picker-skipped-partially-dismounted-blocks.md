# BUG-260511.2: Grind picker skipped blocks at integrity 0 with components remaining

## Status: Done
## Severity: Medium
## Version: v2.5.4
## Found In: `NanobotSystem.Grinding.cs` (ServerTryGrinding picker + BUG-165 drop), `NanobotSystem.Update.cs` (IsLiveGrindTarget)

## Description

Player report: when grinding ships, some blocks (especially wheels and other
attachable / subgrid-component blocks) ended up with integrity at 0 % but never
got fully ground — they remained on the grid (or floated free as detached
subgrids) with components still in the construction stockpile.

## Root cause

`ServerTryGrinding` picker filter at line 93 was:

```csharp
if (!targetData.Block.IsDestroyed)
{
    needGrinding = true;
    chosenGrindTarget = targetData;
    break;
}
```

`IsDestroyed` in SE is true as soon as `Integrity <= 0`. But a block can be
`IsDestroyed && !IsFullyDismounted` — integrity reached 0 (often from damage
or partial grind) **with components still in the construction stockpile**.
The picker skipped those (incrementing `skippedByDestroyed`), so the BaR never
called `ServerDoGrind` to strip the remaining components, the block never
reached `IsFullyDismounted = true`, `RazeQueueHandler` never enqueued it
(it requires the block to be fully razed/dismounted to actually call
`RazeBlocks`), and the block stayed on the grid forever.

Same wrong filter in two more places:

- `Grinding.cs:142` — BUG-165 drop-from-target-list optimization, would
  prematurely remove integrity-0 blocks before their components were stripped.
- `Update.cs` `IsLiveGrindTarget` — the targets-overlay (FEAT-260511.3) filtered
  the same way, hiding the orphaned blocks from the visualization too.

## Fix

Replaced `IsDestroyed` with `IsFullyDismounted` at all three sites:

```csharp
// Picker
if (!targetData.Block.IsFullyDismounted) { … pick … }

// BUG-165 drop
if (grindBlock != null && grindBlock.IsFullyDismounted) { … remove from list … }

// Visualization
if (block.IsFullyDismounted) return false;
```

`IsFullyDismounted` is true only when integrity is 0 **and** the construction
stockpile is empty — i.e., when there really is nothing more to grind. The
RazeQueueHandler's internal `IsDestroyed` check (it requires the block to
actually be destroyed before razing) still gates the actual `RazeBlocks` call.

## Effect on existing flow

- Blocks at integrity 0 with components stay in `PossibleGrindTargets` until
  fully dismounted. The BaR keeps calling `DecreaseMountLevel` on them, which
  strips components from the stockpile without further damaging integrity
  (already 0). Once components are gone → `IsFullyDismounted = true` → enqueued
  for raze → grid cleaned up.
- Fully-dismounted blocks are filtered out by the picker (same as before via
  `IsDestroyed`), so no extra cycles wasted on done blocks.
- Targets overlay now shows partially-dismounted blocks correctly.

No code path now distinguishes integrity from build-level — all grind decisions
flow through `IsFullyDismounted`, which is the SE-side answer to "is anything
left to grind on this block?".

## Related

- BUG-165 (block-removed-from-list optimization) — the drop logic is preserved,
  just with the corrected filter.
- FEAT-260511.3 (targets overlay) — visualization now matches picker behaviour.
