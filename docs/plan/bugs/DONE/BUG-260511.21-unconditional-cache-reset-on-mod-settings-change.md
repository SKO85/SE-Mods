# BUG-260511.21: Generalise the assignment-cache + exhausted-flag reset to ALL mod-settings changes

## Status: Done
## Severity: Refactor (correctness + forward-compat)
## Version: v2.5.4
## Found In: `Mod.cs` (`SettingsChanged`)
## Reported by: Player suggestion

## Motivation

The previous design tied the assignment-cache clear (BUG-260511.20)
and the per-BaR loop-exhausted-flag reset (BUG-260511.14) to a
detected change in `MaxSystemsPerTargetGrid`. That works for the
specific bug it fixes, but it's fragile: any **future** setting that
influences picker decisions (priority lists, safe-zone flag,
weld-options, range, etc.) would need its own detection logic, or
risk leaving stale state that strands BaRs the same way.

## Fix

`Mod.SettingsChanged` now unconditionally:

1. Resets `_grindLoopExhausted` / `_weldLoopExhausted` on every BaR.
2. Calls `BlockSystemAssigningHandler.Clear()` to drop every cached
   claim.

This runs on **every** mod-settings broadcast (admin
`/nanobars config set`, file reload, etc.), not just on
`MaxSystemsPerTargetGrid` changes.

The targeted `ReleaseSurplusLockOnsForLoweredLimit` stays — that one
clears `State.CurrentGrindingBlock` which the generic clear doesn't
touch, and only matters on a **lowered** limit (so the
`GridSystemCount` per-grid counters can drop below the new ceiling).

## Cost

- `BlockSystemAssigningHandler.Clear()`: `ConcurrentDictionary.Clear`,
  O(N) for the current entry count — single-digit microseconds at
  peak (~few hundred entries).
- `ResetLoopExhaustedFlags` per BaR: two field writes per BaR; a
  few hundred microseconds for a 58-BaR cluster.
- Total: well under 1 ms per mod-settings change.

Mod-settings changes are admin-driven, infrequent (handful per
session at most). The cost is negligible relative to the recovery
robustness it provides.

## Tradeoff

Brief disruption: BaRs currently grinding a block momentarily lose
their claim. Their next picker call re-claims the same block (still
in their target list, no longer claimed by anyone after the cache
clear). Worst case, another BaR's picker races ahead and claims the
same block first, causing the original grinder to switch to a
different block. This is cosmetic — grinding continues, just maybe
on a different block. The grid-limit picker still enforces
`MaxSystemsPerTargetGrid` correctly.

## Why this is safer than the targeted version

Any settings combination — current or future — leaves a known clean
state after the change. No "did we remember to clear the cache for
this new setting?" maintenance burden. The user's intuition was
right: simplicity wins over targeted-detection logic for an
operation this cheap.

## Forward-compat

If a future setting requires **additional** post-change cleanup
(e.g. clearing a per-grid statistics counter), add it to this same
block in `SettingsChanged`. The generic clear+reset is the canonical
hook.
