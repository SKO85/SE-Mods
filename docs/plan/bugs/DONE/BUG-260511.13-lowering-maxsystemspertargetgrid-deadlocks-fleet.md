# BUG-260511.13: Lowering `MaxSystemsPerTargetGrid` deadlocks the fleet and drops sim

## Status: Done
## Severity: High (functional + sim drop on a routine admin action)
## Version: v2.5.4
## Found In: `Mod.cs` (`SettingsChanged`), `Handlers/GridSaturationHandler.cs`
## Reported by: Player report

## Symptom

Player set `MaxSystemsPerTargetGrid = 4` (was 20) while ~58 BaRs were
grinding 3 large grids. After the change:

- BaR blocks **stop grinding** entirely.
- Sim drops **1.00 → ~0.85** and stays there.

## Root cause

`MaxSystemsPerTargetGrid` is enforced live via `GridSystemCount`
(per-grid live counter, Inc/Dec in the `CurrentWeldingBlock` /
`CurrentGrindingBlock` setters) and `GridSaturationTracker.Rebuild`
which snapshots `GridSystemCount > limit` once per work tick.

Deadlock chain when the limit drops below the existing per-grid count:

1. 58 BaRs each hold `CurrentGrindingBlock` on one of the 3 grids → 3
   `GridSystemCount` entries of ~20 each.
2. Admin lowers limit to 4. `GridSaturationTracker.Rebuild` now marks
   **all 3 grids saturated**.
3. Every BaR's grind picker iterates the full `PossibleGrindTargets`
   list, finds every block on a saturated grid, skips them all
   (`skippedByGridLimit++`).
4. Picker exits without choosing a target → `_grindLoopExhausted = true`.
5. **Nobody releases their lock-on**, so `GridSystemCount` stays at ~20
   per grid. Saturation set never shrinks.

Per-grid count never drops, so no BaR can pick a fresh target. The
fleet is wedged — and because `_grindLoopExhausted` invalidates on
target-list hash changes (every cluster scan ~2 s), each scan triggers
a fresh full-iteration / full-rejection pass across all 58 BaRs.
That's the recurring sim cost.

The setter for `WorkMode` / sort flags (BUG-260511.5 / FEAT-260511.5)
already releases lock-ons on the cluster-key-hash change, but the
mod-wide `MaxSystemsPerTargetGrid` isn't part of any cluster hash, so
no release ever fires.

## Fix

`Mod.SettingsChanged()` now compares the new `MaxSystemsPerTargetGrid`
against a cached `_lastMaxSystemsPerTargetGrid`. When the limit
**drops** and any grid is currently over the new limit, force-release
the lock-ons of every BaR working an over-limit grid:

- Clear `State.CurrentWeldingBlock` and/or `State.CurrentGrindingBlock`
  for matching BaRs (their setters `Dec` `GridSystemCount`).
- Call `BlockSystemAssigningHandler.ReleaseAllForSystem(EntityId)` so
  TTL-cached block claims don't block re-acquisition.

Runs **once**, before the per-BaR `SettingsChanged` fan-out, so the
immediate rescans triggered by per-BaR `SettingsChanged` see a clean
saturation state.

After release:
- `GridSystemCount` drops to whatever survives (typically 0 — every
  over-limit BaR releases).
- Next picker tick: each BaR's grind picker iterates the target list;
  the first `newLimit` BaRs per grid successfully `AssignToSystem` and
  set their `CurrentGrindingBlock` (incrementing the count); the rest
  see the saturated set fill up and skip cleanly.
- Within a few ticks the fleet stabilises with exactly `newLimit` BaRs
  per grid actively grinding; the remainder mark themselves exhausted
  and fast-skip subsequent loops.

## Logging

Logs an Info line when the release fires:

```
MaxSystemsPerTargetGrid lowered to 4; released 54 BaR lock-on(s) across 3 over-limit grid(s).
```

## Scope notes

- Raising the limit (e.g. 4 → 20) does **not** trigger this path — no
  release is needed; saturation lifts naturally on the next Rebuild.
- The cached `_lastMaxSystemsPerTargetGrid` initialises to `-1`, so
  the first `SettingsChanged` after server start never triggers a
  release (no prior limit to compare against).
- `Mod.Settings.DisableLimitSystemsPerTargetGrid` short-circuits all
  limit checks; that path is unaffected.
