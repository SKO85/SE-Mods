# BUG-260501.1: Forced-rescan flag lost when settings-change reshuffles cluster
## Status: Fixed
## Severity: High
## Version: v2.5.4
## Found In: User report — Nearest↔Farthest toggle takes ~10 s+ to visibly switch grind order; profiling session `20260501191840-profiling` (3 large ships, 58-BaR cluster) confirms.

## Description

Toggling a `ClusterRelevantFlags` setting (most user-visible: `GrindNearFirst` / `GrindSmallestGridFirst` / `GrindIgnorePriorityOrder` / `UseIgnoreColor` / `UseGrindColor` / `AllowBuild`) does not promptly switch the cluster's sorted target list. The new sort order surfaces only after `MaxScanSkipDuration = 60 s` elapses or a target gets destroyed and drops `IsTargetListSaturated()` below threshold — far past the ~1-2 s FEAT-080 promised.

## Steps to Reproduce

1. Place ≥2 BaRs that cluster together (same WorkMode, same area overlap).
2. Begin grinding a multi-grid scenario where Nearest-first and Farthest-first produce visibly different first picks.
3. Toggle the GrindNearFirst switch in the terminal of any cluster member.
4. Observe: the BaR continues working blocks in the previous order. New sort order takes effect only when something forces a real scan (block destruction or 60 s timeout).

## Root Cause

`ScanClusterCoordinator.cs:23-29` — `ClusterRelevantFlags` includes `GrindNearFirst`. Toggling it changes the cluster-key hash, so on the next ~2 s `RebuildClusters` tick the toggling BaR is reassigned to a different cluster (one whose other members share the new flag value). The new cluster has a different elected coordinator.

The FEAT-080 trigger (`NanobotSystem.Init.cs:89` → `NanobotSystem.Scanning.cs:40-94` `TriggerImmediateRescan(reason, bypassDebounce: true)`) sets `_rescanForced=true` and zeroes `_lastFullScanTime` on (a) `this` BaR and (b) the **current** coordinator at trigger time. The BUG-139 fix on `this` was meant to survive a reshuffle on the assumption that *if `this` becomes the new coordinator* the flag fires.

That assumption fails when `this` is reshuffled into a cluster where someone else is coordinator. The new coordinator never had `TriggerImmediateRescan` applied. On its next `UpdateSourcesAndTargetsTimer` tick (`Scanning.cs:144-178`):

- `forceRescan = _rescanForced && timeSinceFullScan >= forceDebounce` → `false && X = false`
- All other gate conditions hold (`_InitialScanCompleted`, `!coordExhausted`, `timeSinceFullScan < 60 s`, `IsTargetListSaturated()`)
- → `_scanSkippedSaturated = true; return;` — scan skipped.

With saturation persisting (the cached candidate list still has live blocks), every subsequent tick re-skips. `PreSortClusterCandidates` is the function that publishes the cluster's sorted list and only runs from a real scan, so the pre-toggle (Nearest-first) order keeps being served.

### Profiler evidence (session `20260501191840-profiling`, 30 s)

- `ImmediateRescanTrigger`: 58 calls (trigger fires).
- `AsyncClusterScan`: **3 calls total, all warmup**. Last logged scan at session-start; no further scans for the remaining 13 s after the user toggled at T+17–19 s.
- `SortAndCapGridCandidates`: 2 calls total — no re-sort after the toggle.
- `ScanClusterCoordinator.RebuildClusters`: 14 calls (~every 2 s) — reshuffle path is firing, new coordinators are being elected.
- Cluster has 58 members.

## Fix

Make the rescan signal survive coordinator changes by propagating it inside the function that *causes* coordinator changes — `ScanClusterCoordinator.RebuildClusters`.

After `cluster.ElectCoordinator()` (`Cluster\ScanClusterCoordinator.cs:113`), iterate the cluster's members; if any non-coordinator member carries `_rescanForced=true`, call `coord.InheritForcedRescan()` and clear the member's flag (consumed). New method on `NanobotSystem` sets `_rescanForced=true`, zeroes `_lastFullScanTime` and `_LastTargetsUpdate` on the coordinator so the next 2 s tick immediately enters the scan path with `forceRescan=true`.

Files touched:
- `SKO-Nanobot-BuildAndRepair-System\Data\Scripts\SKO-Nanobot-BuildAndRepair-System\NanobotSystem.Scanning.cs` — add `internal void InheritForcedRescan()`.
- `SKO-Nanobot-BuildAndRepair-System\Data\Scripts\SKO-Nanobot-BuildAndRepair-System\Cluster\ScanClusterCoordinator.cs` — add propagation pass inside the cluster-assignment loop.

Cost: zero on the fast-path skip (line 65) — only the full rebuild path pays. Full rebuild is already O(N); the added inner loop is one bool read per member per cluster, no allocation.

The existing BUG-139 self-flag in `TriggerImmediateRescan` (line 58) is kept — it remains the immediate trigger when no reshuffle happens (`this` is or becomes coordinator). Propagation only runs when reshuffle actually changes coordinators.

## Verification

1. Bump `Constants.BuildId` and rebuild.
2. Repeat the repro session, profiling for 30 s.
3. Expect: `AsyncClusterScan` calls increase noticeably after each toggle (within ~2 s); `SortAndCapGridCandidates`/`PreSortClusterCandidates` re-fire on each toggle; visible grind-order switch happens within a few seconds (limited only by the lock-on of the currently-grinding block).
