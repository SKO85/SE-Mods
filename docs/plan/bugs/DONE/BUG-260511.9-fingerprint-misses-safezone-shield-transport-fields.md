# BUG-260511.9: State fingerprint missed safe-zone, shield, and transport-metadata changes

## Status: Done
## Severity: High (client state divergence — stale HUD gating, beam state)
## Version: v2.5.4
## Found In: `NanobotSystem.Operations.cs` (`ComputeStateFingerprint`)
## Reported by: Codex review (extended audit)

## Description

Codex flagged the missing `SafeZoneAllowsWelding` / `SafeZoneAllowsGrinding`
/ `SafeZoneAllowsBuildingProjections` / `IsShielded` fields as the next
slice of the BUG-150 fingerprint gap (BUG-260511.8 was the lock-on
targets). When a BaR enters or leaves a safe zone / shield bubble while
otherwise idle, only these four bits flip — every list hash and every
working flag stays the same. The fpUnchanged early-return swallowed the
send, clients kept stale HUD gating.

While fixing those four I audited **every** `SyncBlockState` setter
that flips `Changed = true` and compared it to the fingerprint.

| Field | Was hashed | Notes |
|---|---|---|
| `Ready` / `Welding` / `NeedWelding` / `Grinding` / `NeedGrinding` / `NeedCollecting` / `Transporting` | ✅ | flag bits |
| `InventoryFull` / `LimitsExceeded` | ✅ | flag bits |
| `CurrentWeldingBlock` / `CurrentGrindingBlock` | ✅ | fixed in BUG-260511.8 |
| `CurrentTransportTarget` | ✅ | BUG-155 |
| `PossibleWeldTargets` / `PossibleGrindTargets` / `PossibleFloatingTargets` / `MissingComponents` | ✅ | `CurrentHash` |
| `SafeZoneAllowsWelding` / `SafeZoneAllowsGrinding` / `SafeZoneAllowsBuildingProjections` / `IsShielded` | ❌ | **Codex P1** — fixed here |
| `LastTransportTarget` | ❌ | fixed here |
| `CurrentTransportIsPick` | ❌ | fixed here |
| `CurrentTransportTime` / `CurrentTransportStartTime` | ❌ | fixed here |

## Fix

All eight missing fields are now hashed in `ComputeStateFingerprint`.
Bools as 0/1 mixers (same shape as the existing `Ready`/`Welding`/…),
`TimeSpan` as `.Ticks`, nullable `Vector3D` quantised to int metres
(same shape as `CurrentTransportTarget`).

The transport timing fields are seeded **once per transport** (BUG-260511.1
gate), not per simulation tick, so adding them to the fingerprint does
not churn it on every weld/grind iteration.

## Forward-compatibility note

Any future `SyncBlockState` field that flips `Changed = true` must also
be added to the fingerprint, or the BUG-150 fpUnchanged path will
silently drop its changes. The fingerprint and the `Changed` surface
must stay in sync — they are two views of the same "state visible to
clients" set.
