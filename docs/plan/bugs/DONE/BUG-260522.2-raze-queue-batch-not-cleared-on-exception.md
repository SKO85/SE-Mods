# BUG-260522.2: RazeQueueHandler `_batchByGrid` not cleared on exceptional exit

## Status: Done (BuildId 260522.2)
## Severity: Medium (unintended extra `RazeBlocks` calls on the next drain after a thrown exception)
## Version: v2.5.4
## Found In: `Handlers/RazeQueueHandler.cs`
## Reported by: Codex (chatgpt-codex-connector) — PR #136 review comment

## Description

`RazeQueueHandler.Process()` reuses a static `_batchByGrid`
dictionary as scratch space across calls. The previous version cleared
it **only on the normal success path**, inside the
`if (gridsTouched > 0)` block after the inner foreach. If anything
inside the try block threw before reaching that clear — for example
while reading queued block state in the dequeue loop, or computing
`gridsTouched` — the catch handler logged the exception and exited
**without clearing the scratch**.

On the next successful call, those stale grid → position-list entries
would be merged with the fresh batch and replayed as unintended extra
`RazeBlocks` calls (positions that may already be welded, or worse,
positions on grids no longer relevant). Also keeps grid references
alive longer than intended.

## Fix

Move `_batchByGrid.Clear()` into a `finally` block so it runs on
every exit path (normal, caught exception, or any future early-return
path):

```csharp
try
{
    // dequeue loop, then per-grid RazeBlocks foreach
}
catch
{
    // existing exception-profiler log
}
finally
{
    _batchByGrid.Clear();
}
```

The dictionary entries are list references (`Dictionary<IMyCubeGrid,
List<Vector3I>>`); clearing the dictionary releases its hold on those
lists but doesn't free the list buffers — the next iteration allocates
fresh `List<Vector3I>` instances anyway. So no buffer-pool benefit
lost, just correctness gained.

## Cost

`Clear()` on a ~1–5 entry dictionary, sub-microsecond. Runs once per
`Process()` call which is gated by `ProcessIntervalTicks` (every 10
ticks).
