# BUG-260511.18: Debug HUD "BlockAssigns" count climbs unboundedly due to rare cleanup

## Status: Done
## Severity: Low (display only — picker logic was already correct)
## Version: v2.5.4
## Found In: `Managers/PeriodicMaintenanceScheduler.cs`
## Reported by: Player observation while diagnosing grind engagement

## Symptom

> "BlockAssigns in the debug panel kept increasing."

During active grinding the `BlockAssigns` count shown in the debug HUD
appeared to grow without bound, suggesting a leak.

## Root cause

`BlockSystemAssigningHandler.AssignmentCount` returns the raw
`Cache.Count` — the size of the underlying `ConcurrentDictionary`.
`TtlCache.TryGet` correctly filters expired entries at read time, but
`Cache.Count` reflects total entries, including expired ones.

`BlockSystemAssigningHandler.Cleanup` was bundled with the
`_ttlCleanup` group in `PeriodicMaintenanceScheduler`, which runs
once every **2 minutes**. With ~20 active BaRs grinding at ~1 block/s,
~20 new claims per second land in the cache. The TTL is 8 s, so each
claim is logically expired after 8 s — but the entry stays in the
dictionary until cleanup fires, up to 2 minutes later.

Steady-state inflated count: 20 BaRs × 120 s = ~2400 entries displayed
when the actual live count is ~160. Cleanup runs → count snaps down
→ climbs again. Looks like a leak.

The picker logic was never affected — `IsAssignedToOtherSystem` uses
`TryGet`, which filters expired entries, so other BaRs were never
blocked from claiming a block whose stale claim was sitting in the
dictionary post-TTL.

## Fix

Split the assignment cleanup into its own task with a 5 s cadence:

```csharp
private static PeriodicTask _assignmentCleanup = new PeriodicTask
{
    Interval = TimeSpan.FromSeconds(5),
    LastRun = TimeSpan.Zero,
    Work = () => { try { BlockSystemAssigningHandler.Cleanup(); } catch { } }
};
```

Other TTL caches stay in the existing 2 min `_ttlCleanup` group because
their `Count` isn't surfaced anywhere user-visible and they tolerate
stale entries fine.

## Cost

`Cleanup` iterates the dictionary, removes expired keys via a single
pass using the existing `_expiredKeysBuffer`. For ~2400 entries it's
sub-millisecond. Running 5×/min on the background pool — negligible.

## What this does NOT fix

The "1 BaR grinding while others sit idle" symptom is a separate
issue tracked in a follow-up — the cluster scan was returning very
few candidates in the user's session, and the engagement pattern
appears tied to scan throughput, not the assignment cache.
