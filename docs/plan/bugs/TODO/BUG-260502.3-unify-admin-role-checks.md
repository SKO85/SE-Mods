# BUG-260502.3: Unify admin-role checks on direct enum comparison
## Status: Open
## Severity: Low
## Version: v2.5.4
## Found In: REVIEW-260502.1 (full Codex code review), finding #3

## Description

Three sites compare `MyPromoteLevel` via `ToString()` and string equality:

- `Chat/ChatHandler.cs:286-287` (`IsAdmin`)
- `Handlers/HudHandler.cs:667-668` (`IsLocalPlayerAdmin`)
- `Handlers/NetworkMessagingHandler.cs:364-365` (`SendToAdmins`)

A fourth site already does the right thing — direct enum compare with a `// REF-4:` comment establishing the pattern:

- `Handlers/NetworkMessagingHandler.cs:158-175` (`IsRemoteAdmin`)

The string pattern allocates per call (`PromoteLevel.ToString()` boxes), and breaks silently if SE renames an enum member.

## Fix

Replace the three string-compare sites with the canonical pattern:

```csharp
var level = player.PromoteLevel;
return level == MyPromoteLevel.Admin
    || level == MyPromoteLevel.SpaceMaster
    || level == MyPromoteLevel.Owner;
```

Optional: extract a single static helper (e.g. `IsAdminLevel(MyPromoteLevel level)`) shared across the four call sites so future additions stay consistent.

## Verification

`dotnet build` clean. Manual: confirm `/nanobars` admin-only commands still gate on admin/SpaceMaster/Owner, HUD admin overlay still shows for those roles, and `SendToAdmins` still broadcasts to the right players.
