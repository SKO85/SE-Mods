# BUG-260522.4: `NeededComponentsForBlueprint` missing null/empty Components guard

## Status: Open
## Severity: Low (potential NRE inside a scripting-property call path)
## Version: v2.5.4
## Found In: `Utils/UtilsProductionBlock.cs:32`
## Reported by: Copilot (copilot-pull-request-reviewer) — PR #136 review

## Description

`UtilsProductionBlock.NeededComponentsForBlueprint` iterates
`blockDefinition.Components` without guarding against `null` or empty:

```csharp
foreach (var component in blockDefinition.Components)
{
    if (componentList.ContainsKey(component.Definition.Id)) ...
}
```

Other code paths in the mod that touch the same SE API
(`SlimBlockExtensions.GetMissingComponents`) explicitly null-guard the
`Components` array. If any block definition in a projected grid ever
returns null/empty Components, this loop will throw `NullReferenceException`
and break the scripting property call. Caller is reached via the
`BuildAndRepair.NeededComponentsForBlueprint` scripting API, so the
exception bubbles up into the player's programmable block.

## Fix

Add the same guard as the sibling code path:

```csharp
var components = blockDefinition?.Components;
if (components == null || components.Length == 0) continue;
foreach (var component in components) { ... }
```

Also `block.BlockDefinition as MyCubeBlockDefinition` on line 31 can
yield null on exotic block types — defensive cast check would round
out the safety.

## Verification

`dotnet build` clean. Manual: build a projection containing block
types known to have unusual definitions (procedural / scripted) and
call the scripting property — should return a count without throwing.
