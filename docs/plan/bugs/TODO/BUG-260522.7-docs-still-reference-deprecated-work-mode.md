# BUG-260522.7: Docs still reference deprecated/removed `GrindIfWeldGetStuck` work mode

## Status: Open
## Severity: Low (documentation accuracy)
## Version: v2.5.4
## Found In: `docs/pages/Build-and-Repair-System/index.md:38`, `docs/pages/Build-and-Repair-System/Scripting/index.md:52`
## Reported by: Copilot (copilot-pull-request-reviewer) — PR #136 review (one open, one suppressed-low-confidence)

## Description

`WorkModes.GrindIfWeldGetStuck` was deprecated in v2.5.4 — removed
from the terminal combobox (per `Terminal/ComboBoxes.cs`) and
migrated to `WeldBeforeGrind` at runtime (per
`SyncBlockSettings.WorkMode` setter and `BUG-260511.12` clamp logic).
The docs haven't caught up:

1. **`docs/pages/Build-and-Repair-System/index.md` line 38** — main
   feature list still advertises it as a player-selectable option:
   > "Work Modes — choose between Weld Before Grind, Grind Before
   > Weld, **Grind If Stuck**, Weld Only, or Grind Only..."
   A reader looking for "Grind If Stuck" in the terminal will not
   find it.

2. **`docs/pages/Build-and-Repair-System/Scripting/index.md` line 52** —
   the scripting-API doc lists the `WorkMode` enum values as
   `WeldBeforeGrind, GrindBeforeWeld, WeldOnly, GrindOnly` with NO
   mention of `GrindIfWeldGetStuck`. But the enum value is still
   present in `Terminal.cs` (kept for backwards-compat with existing
   programmable-block scripts referencing the integer). A script
   author reading the docs has no idea that the legacy integer is
   silently migrated to `WeldBeforeGrind`.

## Fix

For `docs/pages/Build-and-Repair-System/index.md:38` — remove
"Grind If Stuck" from the bullet:

> "Work Modes — choose between Weld Before Grind, Grind Before Weld,
> Weld Only, or Grind Only..."

For `docs/pages/Build-and-Repair-System/Scripting/index.md:52` — add
a note about the deprecated value:

> "`WorkMode` — the active work mode. Cast to/from the `WorkModes`
> enum (`WeldBeforeGrind`, `GrindBeforeWeld`, `WeldOnly`, `GrindOnly`).
> The legacy `GrindIfWeldGetStuck` value is migrated to
> `WeldBeforeGrind` at runtime; scripts that still write it will get
> silently-corrected behaviour but new scripts should not use it."

Optionally cross-link to the v2.5.4 release notes section on the
deprecation.

## Verification

Site preview / rendered Markdown review. No code or behaviour
change; the runtime migration was already in place from earlier work.
