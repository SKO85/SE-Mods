# BUG-260502.5: Silent runtime exception swallowing in scheduler / network / background queue
## Status: Open
## Severity: Low
## Version: v2.5.4
## Found In: REVIEW-260502.1 (full Codex code review), finding #5

## Description

Several runtime paths use empty `catch { }` blocks that hide all exceptions, making real bugs (NREs, race conditions, engine API quirks) invisible:

- `Managers/BackgroundTaskQueue.cs:84` — wraps every queued background action; a faulting cluster scan or other queued work is silently dropped.
- `Managers/PeriodicMaintenanceScheduler.cs:49-54` — TTL cache cleanups (six handlers).
- `Managers/PeriodicMaintenanceScheduler.cs:65-66` — friendly cache rebuild + profiler flush.
- `Managers/PeriodicMaintenanceScheduler.cs:89` — `TryFire` wrapper around any periodic task.
- `Handlers/NetworkMessagingHandler.cs:381,392,404,415` — incoming network message handlers.

Note: the `catch { }` blocks in `Mod.UnloadData()` (Mod.cs:337-375) are intentional — at session unload, one failing handler should not break the rest of the cleanup chain. Those should stay.

## Fix

Replace each runtime `catch { }` with `catch (Exception ex) { Logging.Instance.Write(Logging.Level.Error, "<context>: {0}", ex.Message); }` (or `ex.ToString()` for stack traces in debug builds). Logging is throttled and gated by log level, so the cost is essentially nil when nothing is throwing.

Out of scope: UnloadData blanket catches stay as-is.

## Verification

Add a forced exception in one periodic task body and confirm the stack now appears in the mod log instead of being silently dropped.
