# BUG-260522.5: `Logging` singleton uses double-checked locking without `volatile`

## Status: Open
## Severity: Low (latent — would require a specific memory-reorder window during first access)
## Version: v2.5.4
## Found In: `Utils/Logging.cs:13`
## Reported by: Copilot (copilot-pull-request-reviewer) — PR #136 review

## Description

The `Logging.Instance` getter uses the classic double-checked locking
pattern:

```csharp
private static Logging _instance;
private static readonly object _instanceLock = new object();

public static Logging Instance
{
    get
    {
        if (_instance == null)
        {
            lock (_instanceLock)
            {
                if (_instance == null)
                    _instance = new Logging(...);
            }
        }
        return _instance;
    }
}
```

`_instance` is **not** declared `volatile`. Under the .NET memory
model, another thread can in principle observe a non-null reference
before the constructor has finished initializing the fields (write
reordering across the lock release). The accessing thread would then
read a partially-constructed `Logging` object — potentially calling
methods on null `_Writer` / `_Cache` / etc.

Practical impact today: `Logging.Instance` is read from many threads
(background scans, the various `Logging.Instance.Write` calls inside
`Mod.AddAsyncAction` workers). The first-access race window is narrow
but real on weakly-ordered architectures (ARM, future). On x86/x64
the strong memory model makes it nearly impossible to hit.

## Fix options

Pick one:

1. **Add `volatile`** to `_instance`:

   ```csharp
   private static volatile Logging _instance;
   ```

   Smallest possible change. Documents intent. The C# spec guarantees
   that volatile writes are not reordered with earlier writes, so any
   reader who sees `_instance != null` also sees the fully-constructed
   object.

2. **`Lazy<Logging>`** — replace the manual pattern entirely with
   `private static readonly Lazy<Logging> _lazy = new Lazy<Logging>(() => new Logging(...));`
   The BCL handles thread-safety. Slightly more allocation but
   bulletproof.

3. **Drop the outer null check** and always take the lock. Cheapest
   on first access but pays a lock on every read.

Option 1 is the surgical fix and matches the existing structure.

## Verification

`dotnet build` clean — the only change is one keyword. Manually
confirm Logging output still appears in the mod log file. There's no
runtime behaviour change beyond the memory-ordering guarantee.
