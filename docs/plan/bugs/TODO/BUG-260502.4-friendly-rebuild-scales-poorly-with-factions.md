# BUG-260502.4: Friendly cache rebuild scales poorly with faction count
## Status: Open
## Severity: Low
## Version: v2.5.4
## Found In: REVIEW-260502.1 (full Codex code review), finding #4

## Description

`FriendlyRelationsHandler.Rebuild` (`Handlers/FriendlyRelationsHandler.cs:117`) is scheduled every 5 seconds (`Managers/PeriodicMaintenanceScheduler.cs:58-67`). It walks `Mod.NanobotSystems` in a nested loop and calls `IMyShipWelder.GetUserRelationToOwner(sourceOwnerId)` for each `(distinct_owner, BaR)` pair.

The outer loop is deduped via `seenOwners.Add(sourceOwnerId)`, so actual cost is `O(distinct_owner_count × N)`, not `O(N²)`:

- 60 BaRs / 1 faction → 60 GetUserRelationToOwner calls per cycle
- 60 BaRs / 10 factions → ~600 calls per cycle
- 60 BaRs / 60 distinct owners (worst) → 3,600 calls per cycle

GetUserRelationToOwner is an SE engine call (low-µs but not free). Runs on a background thread, but accumulated cost on multi-faction servers is non-trivial.

## Fix Options

Pick one — **don't combine**:

1. **Lower frequency** — bump the periodic interval from 5 s to 15 s when no faction-relation events fired. Trivial.
2. **Dirty-bit trigger** — hook `MyAPIGateway.Session.Factions.FactionStateChanged` (and BaR add/remove) to set a dirty flag; rebuild only when dirty. Most surgical.
3. **Cache `seenOwners` across rebuilds** — only rebuild for owners new since last rebuild + invalidate when faction state changes.

Option 2 is the right long-term fix; option 1 is the cheap immediate one.

## Verification
