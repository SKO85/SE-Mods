# FEAT-260511.5: Release all BaR assignments on sort-relevant settings change

## Status: Done
## Version: v2.5.4
## Found In: `NanobotSystem.Init.cs` (`SettingsChanged`), `Handlers/BlockSystemAssigningHandler.cs`

## Description

When an admin toggled a sort-affecting setting on a BaR (Grind Nearest First /
Farthest First, Smallest Grid First, priority lists, search mode, etc.), the
new target ordering took effect on the next scan (FEAT-080), but the BaR
continued working its previously-claimed block until it completed — and other
BaRs whose claims were still alive in `BlockSystemAssigningHandler` (default
TTL 2-8 s) couldn't be reassigned.

User-visible: "I toggled Nearest First and BaRs are still chipping away at far
blocks for several seconds. Some BaRs haven't even started grinding their
claimed blocks but won't move on."

Reproduction path was the multi-action wrapper assigning blocks that didn't
end up being the BaR's final `CurrentWeldingBlock` / `CurrentGrindingBlock` —
those phantom claims persisted until TTL expiry.

## Implementation

`SettingsChanged` now:

1. Computes the cluster-key hash **before** and **after** the change (uses
   `_lastClusterKeyHash`, written by `ScanClusterCoordinator.RebuildClusters`
   — represents the hash at last rebuild, i.e. before any setting change).
2. If the hash differs (a *sort-relevant* flag changed; cosmetic settings
   like SoundVolume or ShowArea don't appear in the hash, so they don't
   trigger this):
   - Clears `State.CurrentWeldingBlock` and `State.CurrentGrindingBlock`.
   - Calls `BlockSystemAssigningHandler.ReleaseAllForSystem(_Welder.EntityId)`
     — drops every block this BaR has claimed, not just the lock-on.

```csharp
if (_lastClusterKeyHash != 0 && newClusterHash != _lastClusterKeyHash)
{
    State.CurrentWeldingBlock = null;
    State.CurrentGrindingBlock = null;
    if (Mod.Settings.AssignToSystemEnabled)
    {
        BlockSystemAssigningHandler.ReleaseAllForSystem(_Welder.EntityId);
    }
}
```

The `_lastClusterKeyHash != 0` guard skips the first-time call (no previous
state to compare against).

## Helper added

`BlockSystemAssigningHandler.ReleaseAllForSystem(long systemId)` — walks the
TTL cache once, drops every entry whose value equals the given system id.
ConcurrentDictionary enumerator tolerates concurrent removal, so a single
pass is safe.

## Scope is narrow on purpose

Scoped to the cluster-relevant flag set (`ScanClusterCoordinator.ClusterRelevantFlags`):
`UseIgnoreColor`, `UseGrindColor`, `AllowBuild`, `GrindSmallestGridFirst`,
`GrindNearFirst`, `GrindIgnorePriorityOrder`, plus `WorkMode`, `SearchMode`,
priority strings, owner, conveyor flag, safe-zone state. Cosmetic toggles
(SoundVolume, ShowArea, multipliers) are *not* in the hash and don't trigger
the reset — toggling them doesn't churn in-flight work.

Other BaRs' claims on the blocks this BaR released TTL out naturally over
the next 2-8 seconds (controlled by `AssignmentTtlSeconds`).
