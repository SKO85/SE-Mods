# FEAT-260511.7: `--global` flag for `/nanobars config save` and `config delete`, plus reload source reporting

## Status: Done
## Version: v2.5.4
## Found In: `Chat/Commands/ConfigCommand.cs`, `Models/SyncModSettings.cs`

## Description

Restore parity with the original mod's legacy commands:

- `-cwsf` (create world settings file) — write `ModSettings.xml` to the **world** storage folder.
- `-cpsf` (create PC-side settings file) — write `ModSettings.xml` to the **PC-wide** mod storage folder (`%AppData%\SpaceEngineers\Storage\<mod>`).

The PC-wide file acts as a per-machine default that applies to every world that doesn't have its own world-storage file. `SyncModSettings.Load()` already prefers the world file over the local file, so layering is automatic.

## Commands

| Command | Effect |
|---|---|
| `/nanobars config save` | Writes to world storage (unchanged default). |
| `/nanobars config save --global` | Writes to PC-wide local storage. |
| `/nanobars config create` | Alias for `save` — accepts `--global` too. |
| `/nanobars config delete` | Deletes the **world** file only (was: deleted both). |
| `/nanobars config delete --global` | Deletes the PC-wide file only. |
| `/nanobars config delete --all` | Deletes both files (previous default behavior). |

## Implementation

`SyncModSettings.Save(settings, world)` already supported both targets via `WriteFileInWorldStorage` / `WriteFileInLocalStorage`. The only changes are in `ConfigCommand.cs`:

1. `ExecuteSave(args)` / `ExecuteDelete(args)` parse `--global` (and `--all` for delete) from the args.
2. `HasFlag(args, "--global")` helper scans args[2..] for the flag.
3. `GetSettingsFilePath(bool world)` returns the resolved world-storage path or a `%AppData%\SpaceEngineers\Storage\<mod>` descriptor (SE sandbox doesn't grant `Environment.GetFolderPath`, so the global path is shown symbolically rather than resolved).
4. Help text updated for both `save` and `delete`.

## Reload source reporting

`SyncModSettings.Load()` already preferred the world file over the local file (whichever it finds first is loaded whole, no merge). Added a `SettingsLoadSource` enum (`None` / `WorldStorage` / `LocalStorage`) and a static `SyncModSettings.LastLoadSource` field that records which path the most recent `Load()` actually read from.

`/nanobars config reload` now reports the resolved source in its success message:

- `Settings reloaded from: world storage (ModSettings.xml)`
- `Settings reloaded from: PC-wide local storage (ModSettings.xml — global default; no world file present)`
- `Settings reloaded from: built-in defaults (no ModSettings.xml found in world or PC-wide storage)`

Admins running `--global` save on a machine that already has a world file can now confirm at a glance that the world file is still winning.

## Breaking change note

`/nanobars config delete` no longer wipes both files unconditionally. The new default is world-only, which is the safer semantics. Admins who want the old behavior should add `--all`.
