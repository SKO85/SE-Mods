# FEAT-260511.3: `/nanobars debug targets` visualization

## Status: Done
## Version: v2.5.4
## Found In: `Chat/ChatHandler.cs`, `Handlers/HudHandler.cs`, `Handlers/BlockSystemAssigningHandler.cs`, `NanobotSystem.Update.cs`

## Description

Companion to FEAT-260511.2: shows what each BaR is about to act on. Useful for
diagnosing "is this block actually in any BaR's target list?", "which cluster
claimed this block?", and "how many of my targets are currently assigned vs
just discovered?".

## Implementation

New chat command `/nanobars debug targets`. Admin only, client-side only.
State stored as `HudHandler.LocalTargetsVisible`.

For each enabled BaR's `State.PossibleWeldTargets` and `State.PossibleGrindTargets`,
draws each live target block:

- **Border:** cluster-coloured wireframe (same palette as cluster-area),
  drawn with `BlendTypeEnum.PostPP` so it's visible through other blocks.
- **Fill:** for *assigned* blocks (`BlockSystemAssigningHandler.TryGetAssignedSystem`
  returns true), a semi-transparent red solid fill drawn with `PostPP` so it
  paints over the block's mesh instead of Z-fighting against it. *Unassigned*
  blocks get no fill — wireframe only.

Effect: a grid where every target is filled red = every block has a BaR working
on it. Wireframe-only blocks = visible to the cluster but not yet claimed.

### Stale-target filtering

Target lists are rebuilt every `TargetsUpdateInterval` (~10s), so between
scans they contain blocks that have already been completed. `IsLiveWeldTarget`
and `IsLiveGrindTarget` filter each candidate at draw time:

```csharp
private static bool IsLiveWeldTarget(IMySlimBlock block)
{
    if (block.CubeGrid == null) return false;
    if (block.FatBlock != null && block.FatBlock.Closed) return false;
    if (block.IsProjected()) return true;
    return !block.IsFullIntegrity;
}
```

So the visualization tracks live work, not stale entries.

### Per-frame dedup

Many BaRs typically share the same blocks in their target lists (the scanner
adds a block to every BaR whose AABB sees it). Without dedup the same wireframe
would be drawn N times per frame, alpha-stacking into a near-opaque fill.

```csharp
private static readonly HashSet<BlockSystemAssigningHandler.BlockKey> _drawnTargetsThisFrame;
private static int _drawnTargetsFrame = -1;

// First call each frame clears the set; subsequent BaRs check + add.
ClaimTargetForDraw(block);  // returns false if already drawn this frame
```

`BlockKey` is `(GridEntityId, Vector3I Position)` — the same key
`BlockSystemAssigningHandler` uses, so identical semantics for "the same
physical block".

## Helper added

`BlockSystemAssigningHandler.TryGetAssignedSystem(this IMySlimBlock, out long systemId)`
— reads the TTL-cache entry without modifying it. Used to decide fill-vs-no-fill
in the target overlay.

## Performance

Off by default; same `!IsDedicated` guard as cluster-area. When on, each BaR
iterates its own target lists once per frame. Dedup set is static and reused
across frames — allocation-free at steady state.

On a dedicated client (where `State.Possible*Targets` is empty server-state),
the toggle draws nothing — meaningful on listen-server / single-player only.
