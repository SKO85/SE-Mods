# FEAT-260511.6: Multi-action wrapper for weld / grind operations

## Status: Done (Disabled by default — `MaxActionsPerCycle = 1`)
## Version: v2.5.4
## Found In: `NanobotSystem.Operations.cs` (`MultiWeld`, `MultiGrind`)

## Description

Each BaR's `ServerTryWelding` / `ServerTryGrinding` originally did exactly one
weld/grind per Update10 cycle. With the per-tick `PerTickBudget` (count + ms)
as the only throughput throttle, a fleet of N BaRs could collectively do at
most N actions/cycle — which capped the system below the budget on quiet
target lists (one BaR picks one block per cycle, leaving the budget
under-utilised when only a few BaRs have work).

The multi-action wrapper lets a single BaR claim multiple slots from the
per-tick budget when its target list has them, so the budget is filled even
when only a few BaRs have visible work.

## Implementation

`Operations.cs` switch dispatch wraps `ServerTryWelding` / `ServerTryGrinding`
in a small loop:

```csharp
private void MultiGrind(ref bool grinding, ref bool needGrinding, ref bool transporting, ref IMySlimBlock currentGrindingBlock)
{
    for (int i = 0; i < MaxActionsPerCycle; i++)
    {
        bool g, ng, t;
        IMySlimBlock cgb;
        ServerTryGrinding(out g, out ng, out t, out cgb);
        if (g) grinding = true;
        if (ng) needGrinding = true;
        if (t) transporting = true;
        if (cgb != null)
        {
            currentGrindingBlock = cgb;
            State.CurrentGrindingBlock = cgb;  // commit between iterations
        }
        if (!g) break;
    }
}
```

`State.Current*Block` is committed *between* iterations so the next
iteration's `IsGridOverSystemLimit` check sees the updated `Mod.GridSystemCount`.
(Without that commit, a BaR could touch grid G in iter 1 and grid H in iter 2
without G's count ever being incremented for the work done there.)

`MultiWeld` is symmetric.

## Current setting: `MaxActionsPerCycle = 1`

Despite the commit-between-iterations safeguard, the multi-action path
exposes a structural problem with `MaxSystemsPerTargetGrid`: when a BaR
works on grid G in iter 1 (count[G]+=1) and grid H in iter 2 (count[G]-=1
via the setter, count[H]+=1), the net effect is that **grid G is
undercounted** for the work the BaR just did there. A different BaR can
then sneak into the apparent G slot.

With 58 BaRs all multi-acting, this caused the per-grid limit to be silently
bypassed — admins reported all 58 BaRs welding a single grid when the limit
was 20.

Until a proper fix lands (likely: track "any grid this BaR touched this cycle"
rather than "the last grid this BaR touched"), `MaxActionsPerCycle` is set
to **1**. The wrapper structure stays — re-enabling is a one-line change —
but each BaR does exactly one action per cycle, exactly as before.

Throughput is now paced entirely by the per-tick budgets, both of which are
configurable (`MaxGrindsPerTick`, `MaxGrindMsPerTick`, weld twins) so admins
can raise the ceiling directly without needing multi-action.

## Open follow-up

Implement per-cycle "grids touched" tracking, so the limit applies to "did
this BaR weld/grind on grid G at any point this cycle" rather than "where
is this BaR's `CurrentBlock` right now". Once done, `MaxActionsPerCycle`
can be raised again.
