# FEAT-260511.1: Configurable per-tick ms budgets for weld / grind work

## Status: Done
## Version: v2.5.4
## Found In: `Mod.cs`, `Managers/PerTickBudget.cs`, `Models/SyncModSettings.cs`, `Chat/Commands/ConfigCommand.cs`

## Description

`Mod._grindBudget` and `Mod._weldBudget` (PerTickBudget instances) had a hardcoded
`MaxGrindMsPerTickDefault = 8.0` / `MaxWeldMsPerTickDefault = 8.0` ms wall-clock
cap. With many BaRs (~58) firing in the same tick, the time cap typically wins
over the count cap — total throughput sits at 8 ms / avg-grind-cost, which is
roughly 16 grinds per tick regardless of how high `MaxGrindsPerTick` is set.
Admins reported seeing only ~16 BaRs actively grinding while the rest sat idle.

The count caps (`MaxGrindsPerTick`, `MaxWeldsPerTick`) were already
configurable; the ms caps weren't.

## Implementation

### `Managers/PerTickBudget.cs`

Added a third constructor overload that takes the ms cap as a `Func<double>`
resolver, parallel to the existing `Func<int>` count resolver:

```csharp
public PerTickBudget(Func<int> maxResolver, Func<double> maxMsResolver)
{
    _maxResolver = maxResolver;
    _maxMsResolver = maxMsResolver;
}
```

`TryClaim` reads the resolver each call so admins can re-tune without
restarting the session:

```csharp
var maxMs = _maxMsResolver != null ? _maxMsResolver() : _maxMs;
if (maxMs > 0 && _msThisTick >= maxMs) return false;
```

The original `(int, double)` constructor is kept for the static
mechanical-grind / dismount / projBuild budgets that don't need runtime
tunability.

### `Models/SyncModSettings.cs`

Two new `int` properties (ProtoMember 42 / 43), default 8, range 1-100:

- `MaxGrindMsPerTick` — wall-clock budget for grind work per tick.
- `MaxWeldMsPerTick` — same for weld.

Validated in `CheckLimits`. Persisted to/loaded from `ModSettings.xml`.

### `Mod.cs`

`_grindBudget` and `_weldBudget` switched to the dynamic-cap constructor:

```csharp
private static readonly PerTickBudget _grindBudget =
    new PerTickBudget(GetEffectiveMaxGrindsPerTick, GetEffectiveMaxGrindMsPerTick);
```

New resolvers fall back to the consts when the setting is zero or unset.

### `Chat/Commands/ConfigCommand.cs`

`/nanobars config set MaxGrindMsPerTick <value>` and the weld twin now work,
validated to 1-100.

## Notes

- The defaults (8 ms) are unchanged; an admin who doesn't touch the new
  settings keeps the existing behaviour.
- Recommended starting point for fleets > 30 BaRs: `MaxGrindMsPerTick=30`,
  same for weld. Roughly 4× the previous throughput ceiling, still well within
  a 16.7 ms 60 fps budget for a single tick.
