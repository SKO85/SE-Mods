# FEAT-260511.2: `/nanobars debug cluster-area` visualization

## Status: Done
## Version: v2.5.4
## Found In: `Chat/ChatHandler.cs`, `Handlers/HudHandler.cs`, `NanobotSystem.Update.cs`, `Cluster/ScanClusterCoordinator.cs`

## Description

Admins diagnosing "why is BaR X idle when it should reach grid Y" had no way to
see which BaRs share a cluster or what the cluster actually covers. The
information lives entirely server-side in `ScanClusterCoordinator._clusters`
plus per-BaR `Settings.CorrectedAreaBoundingBox`, but there's no in-game
visualization.

## Implementation

New chat command `/nanobars debug cluster-area` (alias `clusterarea`). Admin
only, client-side only — rejected on dedicated servers because the draw call
runs on the local renderer. State stored as `HudHandler.LocalClusterAreaVisible`.

Drawing logic in `NanobotSystem.UpdateBeforeSimulation`:

- Lead election by **min `Welder.EntityId`** among same-cluster BaRs — same
  rule the server uses in `ScanCluster.ElectCoordinator`, so the BaR that
  draws is the coordinator. No need to sync server state.
- Cluster membership recomputed locally from `ScanClusterCoordinator.ComputeClusterKeyHash`;
  only `IsClusterEligible` BaRs (enabled / functional / ready) participate,
  matching the server's `RebuildClusters` filter.
- Solo clusters (memberCount < 2) skipped — the per-BaR `ShowArea` setting
  already covers them.

Per cluster, the lead draws:

1. **One block-sized wireframe per member**, oriented to the parent grid
   (not welder-local Y), sized from `SlimBlock.Min/Max` so multi-cube blocks
   are outlined correctly. All same cluster colour.
2. **One coordinator marker** at the lead BaR's block — same wireframe plus
   an upward extension along grid-up by `CoordinatorMarkerHeightExtra = 2.5 m`,
   so the coordinator reads as a tall pillar above its block. Same cluster
   colour. Drawn with `BlendTypeEnum.PostPP` so it's visible through other
   blocks.
3. **One per-member working OBB** (oriented to the welder's `WorldMatrix`,
   shape from `Settings.CorrectedAreaBoundingBox` + `CorrectedAreaOffset`).
   Same colour. Depth-tested.

Colour comes from an 8-entry fixed palette keyed by hash % 8: yellow, pink,
green, purple, cyan, orange, red, white. `ClusterPalette` and
`ClusterPaletteNames` arrays kept in lock-step; `GetClusterColorName` exposes
the name for chat output.

On toggle-on, a chat line summarises cluster sizes:
```
4 cluster(s) · 47 BaRs total · sizes: [pink=20, yellow=12, cyan=9, green=6]
```
So the admin can match what they see in 3D to numbers.

## Helper added

`ScanClusterCoordinator.IsClusterEligible(NanobotSystem)` — single source of
truth for "is this BaR a cluster participant?". Called from `RebuildClusters`
on the server *and* `DrawClusterAreaIfLead` on the client, guaranteeing the
visualization can't disagree with the actual clusters.

## Performance

Off by default; the per-frame check in `UpdateBeforeSimulation` is a single
bool read when the flag is false. When on, each BaR does an O(N) walk to
determine lead status; with 58 BaRs that's well under 1 ms/frame. Drawing
itself is N × `MySimpleObjectDraw.DrawTransparentBox` calls per cluster.

Strict `!IsDedicated` guard on the draw — zero cost on a DS.
