## Findings

### 1) High: Safe-zone action state fails open on exception

- Category: Correctness, protection bypass risk
- Location:
  - Handlers/SafeZoneHandler.cs:449
  - Handlers/SafeZoneHandler.cs:453-455
  - Handlers/SafeZoneHandler.cs:475
- Detail:
  - GetActionsAllowedForSystem initializes all permissions to true.
  - On exception, the method logs and returns the permissive defaults.
- Impact:
  - A transient failure in safe-zone evaluation can allow welding, grinding, and projection building when restrictions should apply.

### 2) High: IsProtectedFromGrinding fails open and caches unsafe result

- Category: Correctness, protection bypass risk
- Location:
  - Handlers/SafeZoneHandler.cs:48
  - Handlers/SafeZoneHandler.cs:488
  - Handlers/SafeZoneHandler.cs:589-592
- Detail:
  - ProtectedFromGrindingCache TTL is 15 seconds.
  - In catch, IsProtectedFromGrinding writes false (not protected) to cache and returns false.
- Impact:
  - A transient exception can permit grinding of protected blocks and persist that unsafe decision for up to the cache TTL.

### 3) Medium: Runtime exceptions are swallowed in key scheduler/queue/network paths

- Category: Reliability, diagnostics
- Location:
  - Managers/BackgroundTaskQueue.cs:84
  - Managers/PeriodicMaintenanceScheduler.cs:64-69
  - Managers/PeriodicMaintenanceScheduler.cs:113
  - Managers/PeriodicMaintenanceScheduler.cs:118
  - Handlers/NetworkMessagingHandler.cs:380
  - Handlers/NetworkMessagingHandler.cs:391
  - Handlers/NetworkMessagingHandler.cs:403
  - Handlers/NetworkMessagingHandler.cs:414
  - Mod.cs:541
- Detail:
  - Several runtime paths use empty catch blocks.
  - Faults can disappear silently in background work, periodic maintenance, safe-zone refresh, and debug/profile message handling.
- Impact:
  - Hidden failures can cause inconsistent behavior and make production troubleshooting difficult.

### 4) Medium: Friendly-relations rebuild scales poorly with owner/faction diversity

- Category: Performance, server scaling
- Location:
  - Managers/PeriodicMaintenanceScheduler.cs:76-78
  - Handlers/FriendlyRelationsHandler.cs:123
  - Handlers/FriendlyRelationsHandler.cs:134
  - Handlers/FriendlyRelationsHandler.cs:140
- Detail:
  - Rebuild runs every 5 seconds and performs nested iteration across NanobotSystems.
  - Work is effectively O(distinct_owner_count x system_count).
- Impact:
  - On larger multiplayer servers with many owners/factions, this periodic workload can become a recurring hotspot.

### 5) Low: Admin checks still use PromoteLevel.ToString() in multiple modules

- Category: Consistency, micro-performance
- Location:
  - Chat/ChatHandler.cs:329
  - Handlers/HudHandler.cs:695
  - Handlers/NetworkMessagingHandler.cs:363
- Detail:
  - Role checks rely on string conversion and string compares.
  - Another path already uses direct enum comparisons.
- Impact:
  - Minor allocation overhead and consistency drift risk.

## Non-Finding Notes

- Previously suspected GridSystemCount concurrency/deadlock concerns were not confirmed as major in this pass after source validation and thread-context comments.

## Recommendations

- Fix safe-zone exception behavior first (findings #1 and #2) using fail-closed semantics.
- Replace runtime empty catch blocks in operational paths with contextual error logging.
- Reduce friendly relation rebuild pressure by event-driven invalidation or reduced cadence.
- Unify admin-level checks to direct MyPromoteLevel enum comparisons.
