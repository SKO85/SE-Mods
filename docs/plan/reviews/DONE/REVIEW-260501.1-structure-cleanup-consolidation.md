# REVIEW-260501.1: Structure, cleanup, consolidation, reusable refactor & performance audit
## Status: Done
## Reviewer: Claude (deep audit)
## Date: 2026-05-01
## Version: v2.5.4 (branch fix/peformance-improvement-analysis)

## Scope

Full read-only audit of the mod source tree at
`SKO-Nanobot-BuildAndRepair-System/Data/Scripts/SKO-Nanobot-BuildAndRepair-System/`
focused on five axes:

1. **Structure** — partial-class layout, file size, separation of concerns
2. **Cleanup** — dead/inert code, stale fields, deprecated paths, orphan artefacts
3. **Consolidation** — duplicated logic across welding/grinding/scanning/network/handlers
4. **Reusable refactor** — minimal helper extractions that remove duplication without adding allocations
5. **Performance** — allocations in hot paths, lock scope, redundant engine calls, LINQ residue

The audit covers ~24 K LOC of mod source (excluding `Mods/HudAPIv2.cs` and
`Profiling/MethodProfiler.cs` which are utility/3rd-party). Findings are graded
Low / Medium / High by realistic in-game impact, not by line count touched. Per
project rules, every recommendation is performance-conscious and stays inside
C# v6 / no-`System.Threading` / no-`System.Reflection` constraints.

## Findings

### A. Cleanup — dead, inert and stale code

- **CLN-1 (Med)** `NanobotSystem.Welding.cs:1509-1515` — `IsSameBlock` is dead.
  BUG-131 inlined the lock-on identity check (lines 138-139); no other live
  call site remains. The only other reference is a comment in
  `Models/SyncBlockState.cs:177` describing the concept. Delete the method;
  keep the comment.
- **CLN-2 (Med)** `NanobotSystem.Welding.cs:48-49,203-244,365-366,503,520` —
  `starvedPriorityBits` and `starvedSkipped` are dead instrumentation. The
  bits are declared, reset (line 365) and reported (lines 503/520) but no
  code path ever sets them to non-zero, so the two starved-priority gates at
  lines 203 and 233 never trip. The block-comment at lines 42-49 already
  acknowledges this ("inert under the new control flow"). Remove both fields,
  the two gates and the log placeholders. Saves ~50 lines from the hottest
  method in the mod.
- **CLN-3 (Low)** `NanobotSystem.Welding.cs:1187-1201,1223,1335,1344,1413` —
  `PullSourceStats.TsCanTransfer` and `CanTransferCalls` are kept "for log
  format back-compat" after BUG-148 removed the pre-flight
  `CanTransferItemTo` call. Both are always 0/zero-cost; the `++` at 1413
  produces a counter equal to `TransferAttempts`. Drop both fields and the
  matching `canTransferMs=` placeholder in the format string. (Profiler logs
  parse by name; dropping a key is back-compat-safe.)
- **CLN-4 (Low)** `Handlers/HudHandler.cs:525` — `ModeStuck` counter
  increments for `WorkModes.GrindIfWeldGetStuck`, but the live setter in
  `SyncBlockSettings.cs:108,646` migrates that enum value to `WeldBeforeGrind`
  on entry, so `Settings.WorkMode == GrindIfWeldGetStuck` is unreachable at
  read time. Remove the counter or the case branch. The enum value itself
  must stay (save-file back-compat).
- **CLN-5 (Low)** `Handlers/PriorityHandling.cs:113-119` — commented-out
  `GetEnabled(C a)` overload. Either remove the dead block or keep it; it
  has been dead long enough to count as cruft.
- **CLN-6 (Low)** `Handlers/SafeZoneHandler.cs:548-562` — the third
  `targetBlock.OwnerId == attackerBlock.OwnerId` check is unreachable; both
  branches above (lines 517 and 533) already early-return on that condition,
  and no path between them mutates `OwnerId`. Drop lines 548-552 and let
  control fall straight to `GetUserRelationToOwner`.
- **CLN-7 (Low)** `Handlers/SafeZoneHandler.cs:21,232` — the
  `EmptyZoneList` sentinel exists to avoid allocating a per-call empty list
  in `GetSafeZonesInRange`, but callers iterate it with `foreach` regardless.
  Replace the `result == null ? new List<...>() : result` allocation in
  `GetSafeZonesInRange` with the sentinel directly when no match is found
  (already done) and document that callers must not mutate it. Marginal
  win; mostly a code-clarity fix.
- **CLN-8 (Low)** `Helpers/PowerHelper.cs:26-31` — `if (x >= y) return true;
  return false;` collapses to `return x >= y;`. Trivial.

### B. Consolidation — duplicated logic that hurts maintenance

- **CON-1 (High, perf-relevant)** `NanobotSystem.Scanning.cs:753-784,
  870-902, 969-999, 1036-1067` — the
  mechanical / attachable-top / connector child-grid traversal block is
  copy-pasted **four** times across `AsyncAddBlocksOfGrid`'s fast paths and
  the main slow path. Three of those copies (753, 870, 969) also allocate a
  fresh `var fatBlocks = new List<IMySlimBlock>()` per call which thwarts the
  BUG-110 pooling effort.

  Recommended fix (small, no allocations):

  - Extract a static helper
    `TraverseConnections(IMySlimBlock slimBlock, ref MyOrientedBoundingBoxD
    areaBox, ...recursion params...)` with the cast chain and recursion call.
    All four sites become a single line.
  - Add one pooled `_fatBlocksScratch` instance field (mirror of
    `_ScanGridsBuffer` etc.) and reuse it across the three fast paths.
    The slow path at 1036 already iterates `newBlocks` (cache-backed), so
    no allocation there.

  Saves ~120 lines, eliminates 3 per-grid scratch allocations, keeps the
  cast/recursion logic in one place where future changes (e.g. a new
  connection block type) only need to land once.

- **CON-2 (Med)** `Mod.cs:140-225` — `TryClaimGrindSlot` /
  `TryClaimWeldSlot` / `ReportGrindTime` / `ReportWeldTime` /
  `GetEffectiveMaxGrindsPerTick` / `GetEffectiveMaxWeldsPerTick` are
  symmetric pairs differing only in the field/setting names. The existing
  `Managers/PerTickBudget.cs` (used for mechanical/dismount/projBuild) is
  count-only; the weld/grind variants add a `ms` budget.

  Recommended fix: extend `PerTickBudget` with an optional
  `double maxMsPerTick` and a `ReportTime(double ms)` method, then collapse
  the four weld/grind methods into two `PerTickBudget` instances (`_weld`,
  `_grind`) plus a couple of forwarders. ~80 lines of static state + paired
  accessors disappear; peak-tracking moves into the budget class.

- **CON-3 (Med)** `NanobotSystem.Welding.cs:24-402` vs
  `NanobotSystem.Grinding.cs:18-205` — both methods share a near-identical
  structural skeleton (profiler entry, lock acquisition, exhaustion check,
  iteration with priority/grid-limit/assignment/script-controlled gates,
  capture-target-then-defer-work, deferred work outside lock). Welding adds
  lock-on, retry, starved-priority and component pulls; grinding stays
  simpler.

  Full unification is *not* recommended (too many divergent gates, would
  hurt readability and add closure allocations), but the inner per-iteration
  helpers can be reused:

  - Extract `TryClaimAssignmentForBlock(IMySlimBlock, bool isLockOn,
    ref long lastRejectedGridId, out string skipReason)` — body lifts the
    `IsAssignedToOtherSystem` / grid-limit / `AssignToSystem` / failure
    cooldown chain that appears in welding lines 167-272 and grinding lines
    83-103. Both call sites become 4-5 lines instead of 50.
  - Extract `ReleaseAssignmentIfEnabled(IMySlimBlock)` — currently inlined
    7 times in Welding.cs and 3 in Grinding.cs.

  Net effect: each ServerTry method shrinks ~30%, and any future fix to the
  assignment flow lands in one place.

- **CON-4 (Med)** `Handlers/SafeZoneHandler.cs:285-312, 329-340` —
  `GetIntersectingSafeZone` issues `MyAPIGateway.GridGroups.GetGroup` plus a
  `new List<IMyCubeGrid>()` allocation **and then** calls
  `CacheZoneForSubGrids` which does it again with a fresh list. On the
  intersect-found path that is two engine calls + two list allocations per
  zone hit per scan; on the subgrid-intersect path it is up to three.

  Recommended fix: the local `groups` list at line 286 is already populated;
  pass it into `CacheZoneForSubGrids` and have `CacheZoneForSubGrids` skip
  its own GetGroup call. Or pool both lists at handler scope (single-thread
  call site — `GetIntersectingSafeZone` runs inside the scan's existing per
  block check, already serialised by the per-BaR scan thread). Removes the
  duplicate engine call entirely.

- **CON-5 (Low)** `Mod.cs` `IncrementGridCount` /
  `DecrementGridCount` and `Reset*Stats` / `ReportSyncSent` / etc. follow
  the same "static counter + accessor + reset" pattern. Acceptable as-is,
  but if CON-2's `PerTickBudget` extension lands, the same pattern can host
  the BG-task / sync-sent stats too.

- **CON-6 (Low)** `Handlers/PriorityHandling.cs:69, 160, 173, 186, 197, 204,
  215, 221, 251, 279` — eleven LINQ calls (`Find`, `FirstOrDefault`, `Any`,
  `FindIndex`) on what is itself a `List<PrioItemState<C>>`. None are on the
  hot path (terminal UI / save-load only), so this is a stylistic clean-up,
  not a perf fix. Replace with index-based loops if/when this file is
  touched for another reason.

### C. Reusable refactor — small extractions that pay back

- **REF-1 (High value)** `NanobotSystem.cs:99-103` declares
  `_TempPossibleWeldTargets` / `_TempPossibleGrindTargets` /
  `_TempPossibleFloatingTargets` / `_TempPossibleSources` /
  `_TempPossiblePushTargets` as instance fields, but they are written *only*
  from background scan threads (`Scanning.cs:2369-2716`) and then drained
  into the locked `State.PossibleXxxTargets`. Document that ownership rule
  on the fields, or move the lifetime explicit by assigning them in
  `AsyncClusterScan` / `ApplyClusterResultToSelf` and clearing in `finally`.
  As-is they read like generic temporaries and a future maintainer could
  unwittingly touch them from the main thread.

- **REF-2 (Med)** `NanobotSystem.Scanning.cs:1209-1233` (`CleanupEmptyGridCache`)
  / `Caches/TtlCache.cs:122-141` (`CleanupExpired`) /
  `Handlers/SafeZoneHandler.cs:182-197` (`CleanupStaleZones`) /
  `Handlers/GridOwnershipCacheHandler.cs:241-261` (`InvalidateGrid`) — all
  follow the "two-pass: collect expired keys into a fresh List, then mutate"
  pattern, each allocating its own list. None are sub-µs hot, but
  `CleanupExpired` is called per-cache (5+ caches) every 5-15s.

  Recommended fix: add a static reusable buffer **inside each cache class**
  (the cleanups are serialised at a single call site so no concurrency
  concern). Or accept a caller-supplied buffer parameter:
  `CleanupExpired(List<TKey> reusableBuffer = null)`. Trivial change,
  removes ~10 List allocations per maintenance cycle.

- **REF-3 (Med)** `Profiling/MethodProfiler.cs:99,100,126,292,326,545` —
  `_methodStats.ToList()` / `_gridCosts.ToList()` and friends in the
  profiler emit path. These run only when a profiling session is active or
  finalising, so allocations are acceptable. **However**, the
  `domainAgg.ToList()` / `domainStats.ToList()` are called every emit
  iteration in `BuildDomainSummary` etc. — verify (offset/limit reads
  needed) that this isn't inside a per-method loop. If it is, hoist the
  list out once per call.

- **REF-4 (Low)** `Handlers/NetworkMessagingHandler.cs:153-166`
  (`IsRemoteAdmin`) and `Mod.cs:526-540` (welcome message) both compare
  `player.PromoteLevel.ToString()` against `"Admin"` / `"SpaceMaster"` /
  `"Owner"`. The `.ToString()` allocates per call. Compare the enum value
  directly: `if (level == MyPromoteLevel.Admin || level ==
  MyPromoteLevel.SpaceMaster || level == MyPromoteLevel.Owner)`. Welcome
  fires once per session so no measurable saving, but `IsRemoteAdmin` is
  per chat command — small-but-free win, and the string-compare style is
  fragile (any enum rename breaks it silently).

- **REF-5 (Low)** Format-string boilerplate in
  `NanobotSystem.Welding.cs:687-697,765-773,900-915,950-968` and
  `NanobotSystem.Grinding.cs:336-348,415-425,502-514` — every "early-exit
  with profiler emit" path duplicates the entire format-string. Multiple
  copies must be kept in sync when the format is edited (and BUG-141
  already shows one drift: `canTransferMs` is now always 0). If the
  profiler is not turned on (the common case), `MethodProfiler.Start()`
  returns 0 and the closures never run, so consolidating these would not
  even cost an allocation. A single `EmitWeldProfile(...)` /
  `EmitGrindProfile(...)` helper that takes the variant fields would cut
  ~150 lines of duplication. Risk: small — the format-string is the public
  contract for profile log parsing; pin it to a single source.

### D. Performance — concrete hot-path issues

- **PERF-1 (High)** `NanobotSystem.Scanning.cs:753, 870, 969` — three
  per-grid `new List<IMySlimBlock>()` allocations on every scan call. Each
  cluster scan touches dozens of grids; a 60-BaR server can churn hundreds
  of these lists per scan window. They escape to gen-1 and contribute to
  the GC rhythm noted in BUG-110. Pool a single instance field
  (`_fatBlocksScratch`) and `Clear()` it before each use. This pairs
  naturally with CON-1.

- **PERF-2 (High)** `Handlers/SafeZoneHandler.cs:286-287, 331-332` —
  duplicate `MyAPIGateway.GridGroups.GetGroup` engine calls when a grid
  intersects a zone, plus two `new List<IMyCubeGrid>()` allocations.
  GridGroups walks every grid in the mechanical group; on a 30+ grid base
  this is real work. See CON-4 for the fix.

- **PERF-3 (Med)** `Handlers/GridOwnershipCacheHandler.cs:36, 71-74` — the
  cache uses `MyAPIGateway.Session.ElapsedPlayTime.Ticks` directly instead
  of `Mod.NowPlayTime.Ticks`. Every cache write/touch pays the
  Session+ElapsedPlayTime accessor chain — the same chain `TtlCache`
  already optimises away via `Mod.NowPlayTime`. Switch both call sites to
  the cached value (fall back to live only if `NowPlayTime == TimeSpan.Zero`,
  same pattern as `TtlCache.TryGetNow`).

- **PERF-4 (Med)** `Handlers/GridOwnershipCacheHandler.cs:190, 246` —
  `Cache.Entries.Keys.ToArray()` allocates per call.
  `RefreshExpiredEntries` runs every 5 s (acceptable, ~1 alloc / 5 s).
  `InvalidateGrid` is called from inside `RefreshExpiredEntries`'s loop on
  every stale grid, so the array is reallocated *per stale grid*. Hoist
  the `keys` array out (read once, share across InvalidateGrid calls in
  the same Update tick) or have InvalidateGrid scan the *original* keys
  array passed by reference. Both are simple changes.

- **PERF-5 (Med)** `NanobotSystem.Operations.cs:15-307`
  (`ServerTryWeldingGrindingCollecting`) — at 290 lines in one method, the
  control flow is hard to read but mostly necessary. The work-mode switch
  block at lines 145-173 is a classic candidate for extraction into a
  small dispatch helper, *but* doing so would close over many out-args.
  Keep as-is; only call out the long body to remind future contributors.

- **PERF-6 (Med)** `Handlers/PriorityHandling.cs:298-327` — `UpdateHash`
  rebuilds `_ClassList` (a `MemorySafeList<string>`) by `string.Format`-ing
  each entry on every dirty rebuild. Call frequency is low (terminal /
  setting changes), but the format is `"{key};{enabled}"`, which can be
  built without `string.Format` (`key.ToString() + ";" + (enabled ?
  "True" : "False")`). Same output, no boxing. Trivial.

- **PERF-7 (Low)** `Handlers/PriorityHandling.cs:255-263` (`GetEntries`) —
  string concatenation in a loop with `+=` produces O(n²) intermediate
  strings. Called on save only. Replace with `StringBuilder` if/when
  touched.

- **PERF-8 (Low)** `Handlers/NetworkMessagingHandler.cs:155` — `var players
  = new List<IMyPlayer>()` allocation per chat command. The existing FEAT-059
  pattern (`SendToAdmins` reusable list) could be applied here too. Low
  frequency, low impact, but free if you reuse the same shared buffer.

- **PERF-9 (Low)** `Handlers/SafeZoneHandler.cs:74-76` — `GetSafeZones`
  allocates `new HashSet<IMyEntity>()` per call. Called on register and
  every CleanupSafeZones cycle (6 s). Pool at handler scope.

- **PERF-10 (Low)** `NanobotSystem.cs:64` — `private static readonly
  Random _RandomDelay = new Random();` is shared across all BaR instances
  and accessed from `TryTransmitState` (multiple BaRs may race) and
  `UpdateBeforeSimulation10_100`. `Random` is **not** thread-safe; both
  call sites are main-thread-only today, but the single instance also
  produces correlated jitter values. Per-instance Random seeded from
  `Welder.EntityId` would decorrelate per BaR and remove the contention
  risk if anything ever moves the call off the main thread.

- **PERF-11 (Low)** `NanobotSystem.Welding.cs:540` — `_BrokenProjBuildKeys
  .Count > 0 && _BrokenProjBuildKeys.Contains(...)` — `Contains` already
  short-circuits on empty sets in O(1); the `.Count > 0` guard is a tiny
  redundant read. Defensible micro-optimization or just dead-style. Drop
  the guard or keep it.

### E. Structure — file size and partial-class layout

- **STR-1 (Med)** `NanobotSystem.Scanning.cs` (2 766 LOC) is the largest
  file in the mod and the warning shadow approaches the 3 K-line "split
  candidate" line. The natural seam is between the *cluster
  coordinator/member orchestration* (StartAsyncClusterScan,
  AsyncClusterScan, ApplyClusterResultToSelf, scan timers, idle backoff —
  ~lines 100-650 + 2200-2766) and the *scanner primitives*
  (AsyncAddBlock\* / AsyncAddBlocksOfGrid / AsyncAddBlocksOfBox /
  GetBlocksFromCache / sort+truncate helpers — ~lines 380-2100). Splitting
  into `NanobotSystem.Scanning.cs` (orchestration) and
  `NanobotSystem.Scanning.Block.cs` (primitives) keeps the partial class
  intact and makes navigation tractable. Not urgent; flag for next time
  this file is touched.

- **STR-2 (Low)** `NanobotSystem.Welding.cs` (1 530 LOC) is heavy but the
  partial-class boundary is already correct. The CLN-1/CLN-2/CLN-3
  cleanups bring it under ~1 350 lines, which is comfortable.

- **STR-3 (Low)** `Handlers/HudHandler.cs` (815 LOC) and
  `Models/SyncBlockSettings.cs` (811 LOC) are large but cohesive.
  Acceptable.

### F. Documentation — consistency drift between code and CLAUDE.md

- **DOC-2 (Low)** `CLAUDE.md` "Key Files" section lists

## Recommendations

Three batches sized so each is a self-contained PR with profiler verification.

**Batch 1 — Pure cleanup (no behavioural change, low risk).**
CLN-1, CLN-2, CLN-3, CLN-4, CLN-5, CLN-6, CLN-8, PERF-11, DOC-2.
Net effect: ~150 fewer lines, same behaviour. Single profile-session
before/after to confirm no regression.

**Batch 2 — Performance wins (small, measurable).**
CON-1 + PERF-1 (paired — extract `TraverseConnections` and pool
`_fatBlocksScratch`), CON-4 + PERF-2 (paired — pass `groups` to
`CacheZoneForSubGrids`), PERF-3, PERF-4, REF-2, REF-4, PERF-6, PERF-9.
Profile a busy server (50+ BaRs / 10K-block bases) before/after; expect
visible drops in scan and SafeZoneHandler entries.

**Batch 3 — Consolidation (larger, behavioural-equivalent refactor).**
CON-2 (extend PerTickBudget), CON-3 (extract assignment helpers from
weld/grind loops), REF-5 (single profiler emit per loop variant), STR-1
(split Scanning.cs). Land last because diff size is largest and risk of
introducing a subtle regression is highest. Each finding still small
individually; the pile-up is what makes this a separate batch.

**Out of scope intentionally** (still): STR-2/STR-3 (file-size cosmetic),
CON-6 (LINQ on cold paths). Punt unless touched for another reason.
*(PERF-7, PERF-10 were originally in this list but were rolled into the
LOW-severity sweep on 2026-05-01 — see Implementation log below.)*
